Rôle 3 — Compute Engineer — TP05 Nextcloud

Durée : 6h (journée complète) Module livré : modules/compute/ % de la note : 20 %

🔵 Starter kit : vous démarrez depuis corriges/jour5/tp05-starter/ qui contient déjà l'arborescence, les interfaces figées (variables.tf + outputs.tf) et les providers. Vous remplissez les fichiers .tf vides avec des TODO.

Votre mission

  • Construire l'étage applicatif de Nextcloud : ALB public HTTPS + Auto Scaling Group d'EC2 privées qui lancent le container Docker Nextcloud au boot.
  • Générer un certificat auto-signé via le provider hashicorp/tls et l'importer dans ACM pour que l'ALB puisse écouter en HTTPS sans acheter de domaine.
  • Produire un user_data templatifié qui installe Docker, lit les secrets via Secrets Manager (IAM instance profile fourni par le Rôle 5) et démarre le container nextcloud:30-apache branché sur RDS + S3.
  • Exposer les outputs figés du contrat : alb_dns_name, alb_zone_id, asg_name, nextcloud_url — que le Rôle 1 remontera dans envs/dev/outputs.tf.
  • Activer les access logs ALB vers le bucket S3 créé par le Rôle 4 (vous consommez son s3_logs_bucket_name, vous ne créez pas le bucket).

🟤 Contexte Kolab : le cabinet d'avocats ouvre Nextcloud uniquement en interne depuis le VPN du cabinet. Pour la formation on reste sur une ALB 100 % publique en HTTPS self-signed — les avocats cliqueront "Accepter le risque" une fois, point. Le scaling horizontal n'est pas demandé (sans Redis en cache de session, Nextcloud ne tolère pas 2 frontaux simultanés), l'ASG sert uniquement à l'auto-recovery si l'EC2 crash.

🔵 Rappel concept : ALB + target group + listener

Un Application Load Balancer est un load balancer HTTP/HTTPS managé par AWS. Il écoute sur un port public (443), termine le TLS, et route les requêtes vers un target group (pool d'EC2). On configure 4 ressources :

  • aws_lb : le load balancer lui-même (dans les subnets publics)
  • aws_lb_target_group : le pool de cibles avec son health check
  • aws_lb_listener : la règle d'écoute sur un port (443 HTTPS, 80 redirect)
  • aws_acm_certificate : le certificat TLS (ici self-signed pour le TP)

🔵 Rappel concept : Launch Template + Auto Scaling Group

Un launch template est la "recette" pour fabriquer une EC2 (AMI, instance type, user_data, SG, IAM profile). Un ASG (Auto Scaling Group) l'utilise pour maintenir N instances vivantes entre min_size et max_size. Si une EC2 crash, l'ASG en recrée une automatiquement. Ici : min=1, max=2, desired=1 — auto-recovery, pas de scaling horizontal.

🔵 Rappel concept : user_data avec templatefile()

Le user_data est un script shell exécuté une seule fois au premier boot de l'EC2 (via cloud-init). On peut y interpoler des variables Terraform via la fonction templatefile() :

user_data = base64encode(templatefile("${path.module}/templates/nextcloud.sh.tftpl", {
  db_endpoint            = var.db_endpoint
  db_password_secret_arn = var.db_password_secret_arn
}))

Dans le template : ${db_endpoint} est remplacé par la valeur Terraform. Pour garder une variable bash littérale ($VARIABLE), il faut échapper en $${VARIABLE}.

🔵 Rappel concept : certificat self-signed via provider tls

En prod on achète un vrai cert via ACM + validation DNS. Pour ce TP, on génère un cert auto-signé en mémoire Terraform (via le provider hashicorp/tls), puis on l'importe dans ACM pour que l'ALB puisse l'utiliser. Le navigateur affichera un avertissement "Connexion non privée" — on clique "Continuer".

🔵 Rappel concept : IAM Instance Profile (côté consommation)

L'EC2 Nextcloud a besoin d'accéder à S3 (stockage fichiers) et Secrets Manager (passwords) sans aucune clé AWS en dur. Pour ça, elle "porte" un IAM role via un instance profile (fourni par le Rôle 5). Au boot, le SDK AWS récupère automatiquement des credentials éphémères via l'IMDS (Instance Metadata Service). Vous consommez var.app_instance_profile_name dans votre launch template.

Étape 2 : versions.tf et locals.tf (10 min)

Objectif : figer les versions Terraform + provider, et centraliser les valeurs dérivées.

Contexte : le provider tls (non-AWS) est indispensable pour générer le cert auto-signé. C'est un provider que vous n'avez pas encore utilisé, pensez à le déclarer dans versions.tf.

Exemple :

# versions.tf
terraform {
  required_version = ">= 1.10"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.70"
    }
    # Provider TLS pour generer la cle + le cert auto-signe
    tls = {
      source  = "hashicorp/tls"
      version = "~> 4.0"
    }
  }
}

# locals.tf
locals {
  name_prefix = "${var.project_name}-${var.environment}"

  # Transformation map -> list pour les ressources qui attendent une liste
  # (ALB subnets, ASG vpc_zone_identifier).
  public_subnet_ids_list      = values(var.public_subnet_ids)
  private_app_subnet_ids_list = values(var.private_app_subnet_ids)

  common_tags = {
    Project     = var.project_name
    Environment = var.environment
    ManagedBy   = "terraform"
    Module      = "compute"
  }
}

Étape 3 : Certificat auto-signé + import ACM (30 min)

Objectif : avoir un certificat TLS valide (techniquement) pour que l'ALB puisse écouter en HTTPS.

Contexte : en prod, vous achèteriez un domaine et utiliseriez aws_acm_certificate avec validation DNS. Ici on triche : on génère une paire clé/cert dans Terraform, on l'importe dans ACM, et c'est ACM qui expose le cert à l'ALB. Le navigateur affichera un warning "Connexion non privée" — les utilisateurs cliquent "Continuer", c'est accepté pour la formation.

Exemple :

# tls.tf

# 1. Cle privee RSA 2048 bits (generee en memoire Terraform)
# ATTENTION : la cle privee finit dans le state. C est accepte ici car
# le state est chiffre KMS (backend S3 + KMS). En prod pour un vrai
# cert, vous n utiliseriez JAMAIS ce pattern.
resource "tls_private_key" "self_signed" {
  algorithm = "RSA"
  rsa_bits  = 2048
}

# 2. Certificat X509 auto-signe, valide 1 an
resource "tls_self_signed_cert" "alb" {
  private_key_pem = tls_private_key.self_signed.private_key_pem

  subject {
    common_name  = "${local.name_prefix}.kolab.local"
    organization = "Kolab Cabinet Avocats"
  }

  validity_period_hours = 8760  # 365 jours

  # Usages autorises par le cert
  allowed_uses = [
    "key_encipherment",
    "digital_signature",
    "server_auth",
  ]

  # DNS alternatif : on autorise n importe quel domaine ALB AWS
  # (le DNS name de l ALB sera genere apres)
  dns_names = [
    "${local.name_prefix}.kolab.local",
    "*.elb.amazonaws.com",
  ]
}

# 3. Import du cert dans ACM (c est ACM qui parle a l ALB)
resource "aws_acm_certificate" "self_signed" {
  private_key      = tls_private_key.self_signed.private_key_pem
  certificate_body = tls_self_signed_cert.alb.cert_pem

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-alb-cert"
  })

  lifecycle {
    create_before_destroy = true
  }
}

Résultat attendu : terraform plan sur ce fichier seul crée 3 ressources : tls_private_key.self_signed, tls_self_signed_cert.alb, aws_acm_certificate.self_signed.

À vous !

Complétez tls.tf pour que le certificat soit valide 2 ans au lieu d'1 an et inclue en dns_names le nom prévisible de votre ALB (${local.name_prefix}-alb.elb.amazonaws.com en wildcard n'est pas possible — réfléchissez à une alternative via aws_lb.main.dns_name en post-hoc).

Indice 1 : validity_period_hours prend un int, 2 ans = 17520.

Indice 2 : vous ne pouvez pas référencer aws_lb.main.dns_name dans les dns_names du cert car cela créerait un cycle (l'ALB consomme le cert). Solution pragmatique : mettre un wildcard *.elb.amazonaws.com ou juste le common_name.

🔷 Point clé à vérifier : terraform plan affiche bien + aws_acm_certificate.self_signed avec un status qui passera à ISSUED dès apply.

🟢 Voir la solution

Étape 4 : Application Load Balancer (30 min)

Objectif : créer l'ALB public + target group + listeners HTTPS/HTTP.

Contexte : l'ALB vit dans les subnets publics (2 AZ), il reçoit le trafic HTTPS et le forward en clair vers les EC2 privées sur le port 80 (Docker Nextcloud expose du HTTP en interne). Il écrit ses access logs dans le bucket S3 du Rôle 4.

Exemple :

# alb.tf

# -----------------------------------------------------------------------------
# Application Load Balancer (public, cross-AZ)
# -----------------------------------------------------------------------------
resource "aws_lb" "main" {
  name               = "${local.name_prefix}-alb"
  internal           = false
  load_balancer_type = "application"
  security_groups    = [var.alb_security_group_id]
  subnets            = local.public_subnet_ids_list

  # Protection contre destruction accidentelle en prod
  # FALSE en dev pour que terraform destroy fonctionne sans souci
  enable_deletion_protection = false

  # Recommandations AWS / tfsec
  drop_invalid_header_fields = true
  enable_http2               = true

  # Access logs : on ecrit dans le bucket fourni par le Role 4 (Data)
  access_logs {
    bucket  = var.s3_logs_bucket_name
    prefix  = "alb"
    enabled = true
  }

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-alb"
  })
}

# -----------------------------------------------------------------------------
# Target Group : les EC2 Nextcloud enregistrees par l ASG arriveront ici.
# Health check sur /status.php, endpoint natif Nextcloud qui renvoie un JSON.
# -----------------------------------------------------------------------------
resource "aws_lb_target_group" "app" {
  name        = "${local.name_prefix}-tg"
  port        = 80
  protocol    = "HTTP"
  vpc_id      = var.vpc_id
  target_type = "instance"

  # Deregistration delay : temps avant retrait d une EC2 degradee
  deregistration_delay = 30

  health_check {
    enabled             = true
    path                = "/status.php"
    port                = "traffic-port"
    protocol            = "HTTP"
    matcher             = "200"
    interval            = 30
    timeout             = 10
    healthy_threshold   = 2
    unhealthy_threshold = 3
  }

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-tg"
  })

  # create_before_destroy : evite une fenetre sans TG si on renomme
  lifecycle {
    create_before_destroy = true
  }
}

# -----------------------------------------------------------------------------
# Listener HTTPS 443 : c est la porte d entree
# -----------------------------------------------------------------------------
resource "aws_lb_listener" "https" {
  load_balancer_arn = aws_lb.main.arn
  port              = 443
  protocol          = "HTTPS"
  # Politique TLS moderne (TLS 1.2 et 1.3, pas de SSLv3/TLS1.0)
  ssl_policy      = "ELBSecurityPolicy-TLS13-1-2-2021-06"
  certificate_arn = aws_acm_certificate.self_signed.arn

  default_action {
    type             = "forward"
    target_group_arn = aws_lb_target_group.app.arn
  }

  tags = local.common_tags
}

# -----------------------------------------------------------------------------
# Listener HTTP 80 : redirige tout vers 443 (UX + securite)
# -----------------------------------------------------------------------------
resource "aws_lb_listener" "http_redirect" {
  load_balancer_arn = aws_lb.main.arn
  port              = 80
  protocol          = "HTTP"

  default_action {
    type = "redirect"
    redirect {
      port        = "443"
      protocol    = "HTTPS"
      status_code = "HTTP_301"
    }
  }

  tags = local.common_tags
}

🟡 Piège courant : si vous oubliez drop_invalid_header_fields = true, tfsec remonte aws-elb-drop-invalid-headers en HIGH. Le rôle 5 fait un scan tfsec final, ne lui donnez pas de raison de vous renvoyer en PR.

🔷 Point clé à vérifier : le plan affiche + aws_lb.main, + aws_lb_target_group.app, + aws_lb_listener.https, + aws_lb_listener.http_redirect.

Étape 5 : Template user_data Nextcloud (40 min)

Objectif : écrire le script bash qui installe Docker, récupère les secrets, lance Nextcloud.

Contexte : le user_data s'exécute une fois au boot de chaque EC2. Il doit être idempotent (si le boot échoue et qu'AWS relance l'instance, ça doit repartir proprement). On utilise templatefile() pour injecter les valeurs Terraform (endpoints, ARNs) dans le script.

Exemple :

# modules/compute/templates/nextcloud-user-data.sh.tftpl
#!/bin/bash
# User data Nextcloud - execute au boot EC2
# Variables injectees par Terraform via templatefile() :
#   $${db_endpoint}, $${db_name}, $${db_username},
#   $${db_password_secret_arn}, $${admin_password_secret_arn},
#   $${s3_primary_bucket_name}, $${alb_dns_name}, $${aws_region}

set -euxo pipefail

exec > >(tee /var/log/user-data.log | logger -t user-data -s 2>/dev/console) 2>&1

echo "===> Update systeme + install Docker + AWS CLI"
dnf update -y
dnf install -y docker
systemctl enable --now docker
usermod -aG docker ec2-user

# AWS CLI v2 est preinstalle sur AL2023

echo "===> Attente que RDS reponde (jusqu a 5 min)"
for i in {1..30}; do
  if timeout 3 bash -c "cat < /dev/tcp/${db_endpoint}/5432" 2>/dev/null; then
    echo "RDS OK"
    break
  fi
  echo "RDS pas encore pret, retry $i/30..."
  sleep 10
done

echo "===> Recuperation des secrets via IAM instance profile"
DB_PASSWORD=$(aws secretsmanager get-secret-value \
  --region ${aws_region} \
  --secret-id "${db_password_secret_arn}" \
  --query SecretString --output text)

ADMIN_PASSWORD=$(aws secretsmanager get-secret-value \
  --region ${aws_region} \
  --secret-id "${admin_password_secret_arn}" \
  --query SecretString --output text)

echo "===> Pull + run Nextcloud container"
docker pull nextcloud:30-apache

docker run -d --name nextcloud \
  --restart always \
  -p 80:80 \
  -e POSTGRES_HOST="${db_endpoint}" \
  -e POSTGRES_DB="${db_name}" \
  -e POSTGRES_USER="${db_username}" \
  -e POSTGRES_PASSWORD="$DB_PASSWORD" \
  -e NEXTCLOUD_ADMIN_USER=admin \
  -e NEXTCLOUD_ADMIN_PASSWORD="$ADMIN_PASSWORD" \
  -e OBJECTSTORE_S3_BUCKET="${s3_primary_bucket_name}" \
  -e OBJECTSTORE_S3_REGION="${aws_region}" \
  -e OBJECTSTORE_S3_USEPATH_STYLE=false \
  -e OBJECTSTORE_S3_SSL=true \
  -e OBJECTSTORE_S3_AUTOCREATE=false \
  -e OBJECTSTORE_S3_USE_EC2_IAM=true \
  -e NEXTCLOUD_TRUSTED_DOMAINS="*" \
  -e OVERWRITEPROTOCOL=https \
  nextcloud:30-apache

echo "===> Nextcloud lance. Attente 30s pour bootstrap..."
sleep 30
docker ps
echo "===> user-data termine avec succes"

🟤 Notes sur les variables template :

  • $${VAR} = variable bash (n'est pas remplacée par Terraform)
  • ${VAR} = variable Terraform (remplacée par templatefile())
  • NEXTCLOUD_TRUSTED_DOMAINS="*" : accepte n'importe quel Host: header — OK en dev uniquement pour contourner le chicken-and-egg ALB DNS. 🟡 En prod : lister explicitement les domaines.
  • OVERWRITEPROTOCOL=https : force Nextcloud à générer ses URLs internes en https://, sinon il voit le HTTP interne et casse les redirections.

🔷 Point clé à vérifier : le fichier templates/nextcloud-user-data.sh.tftpl existe et ne contient aucun secret en clair — les mots de passe sont tirés via aws secretsmanager get-secret-value à l'exécution.

Étape 6 : AMI data source + Launch Template (30 min)

Objectif : template de machine reproductible avec IMDSv2 forcé, user_data, tags propagés.

Contexte : le Launch Template décrit comment fabriquer une EC2. L'ASG l'utilise pour créer 1 à 2 instances identiques. IMDSv2 obligatoire pour éviter le SSRF — c'est un point que tfsec vérifie.

Exemple :

# asg.tf (partie 1 : AMI + Launch Template)

# -----------------------------------------------------------------------------
# AMI Amazon Linux 2023 (x86_64) la plus recente
# -----------------------------------------------------------------------------
data "aws_ami" "al2023" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-*-x86_64"]
  }

  filter {
    name   = "architecture"
    values = ["x86_64"]
  }

  filter {
    name   = "state"
    values = ["available"]
  }
}

# -----------------------------------------------------------------------------
# Launch Template : recette de construction d une EC2
# -----------------------------------------------------------------------------
resource "aws_launch_template" "app" {
  name_prefix   = "${local.name_prefix}-lt-"
  image_id      = data.aws_ami.al2023.id
  instance_type = var.instance_type

  # IAM instance profile fourni par le Role 5 (permet AWS CLI sur l EC2)
  iam_instance_profile {
    name = var.app_instance_profile_name
  }

  # Security Group app : seul l ALB peut parler HTTP 80 a l EC2
  vpc_security_group_ids = [var.app_security_group_id]

  # IMDSv2 obligatoire (contre le SSRF et les vols de credentials)
  metadata_options {
    http_tokens                 = "required"
    http_endpoint               = "enabled"
    http_put_response_hop_limit = 2  # 2 pour que Docker container puisse acceder IMDS
    instance_metadata_tags      = "enabled"
  }

  # Root volume chiffre par KMS (defaut AWS)
  block_device_mappings {
    device_name = "/dev/xvda"
    ebs {
      volume_size           = 20
      volume_type           = "gp3"
      encrypted             = true
      delete_on_termination = true
    }
  }

  # user_data rendu par templatefile() avec les valeurs des autres modules
  user_data = base64encode(templatefile("${path.module}/templates/nextcloud-user-data.sh.tftpl", {
    db_endpoint                = var.db_endpoint
    db_name                    = var.db_name
    db_username                = var.db_username
    db_password_secret_arn     = var.db_password_secret_arn
    admin_password_secret_arn  = var.admin_password_secret_arn
    s3_primary_bucket_name     = var.s3_primary_bucket_name
    alb_dns_name               = aws_lb.main.dns_name
    aws_region                 = "eu-west-1"
  }))

  # Tags propages aux instances creees par l ASG
  tag_specifications {
    resource_type = "instance"
    tags = merge(local.common_tags, {
      Name = "${local.name_prefix}-app"
      Role = "nextcloud-app"
    })
  }

  tag_specifications {
    resource_type = "volume"
    tags = merge(local.common_tags, {
      Name = "${local.name_prefix}-app-volume"
    })
  }

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-launch-template"
  })

  # Si on change l AMI ou le user_data, on veut un nouveau LT avant suppression
  lifecycle {
    create_before_destroy = true
  }
}

🟡 Piège http_put_response_hop_limit : la valeur par défaut (1) bloque les containers Docker qui veulent lire IMDS. On met 2 pour que Nextcloud puisse utiliser OBJECTSTORE_S3_USE_EC2_IAM=true. Sans ça, Nextcloud n'accède pas à S3 et Google ne vous aidera pas (erreur silencieuse côté Nextcloud).

🔷 Point clé à vérifier : terraform plan sur le LT affiche + user_data = (known after apply) — c'est normal, il dépend de aws_lb.main.dns_name.

Étape 7 : Auto Scaling Group (30 min)

Objectif : l'ASG consomme le LT, maintient 1 instance vivante, s'attache au target group.

Contexte : l'ASG est le "superviseur" — si l'EC2 crash, il en recrée une. health_check_type = "ELB" délègue la décision au target group (si le /status.php renvoie un 5xx, l'ASG tue l'instance).

Exemple :

# asg.tf (partie 2 : ASG)

resource "aws_autoscaling_group" "app" {
  name = "${local.name_prefix}-asg"

  # Subnets prives (les EC2 ne sont JAMAIS exposees directement)
  vpc_zone_identifier = local.private_app_subnet_ids_list

  # Taille : 1 seule instance a la fois (file locking Nextcloud sans Redis)
  min_size         = 1
  max_size         = 2
  desired_capacity = 1

  # Health check ELB : l ASG suit l avis du target group
  health_check_type         = "ELB"
  health_check_grace_period = 300  # 5 min pour que Nextcloud finisse de booter

  # Rattachement au Launch Template (toujours la derniere version)
  launch_template {
    id      = aws_launch_template.app.id
    version = "$Latest"
  }

  # Rolling refresh si le LT change : on cycle les instances une par une
  instance_refresh {
    strategy = "Rolling"
    preferences {
      min_healthy_percentage = 50
      instance_warmup        = 300
    }
  }

  # Tags propages a chaque instance creee (separe du tag_specifications du LT
  # pour etre sur que Name est bien propage).
  tag {
    key                 = "Name"
    value               = "${local.name_prefix}-app"
    propagate_at_launch = true
  }

  tag {
    key                 = "Environment"
    value               = var.environment
    propagate_at_launch = true
  }

  tag {
    key                 = "Project"
    value               = var.project_name
    propagate_at_launch = true
  }

  # create_before_destroy pour zero-downtime si on renomme l ASG
  lifecycle {
    create_before_destroy = true
  }
}

# -----------------------------------------------------------------------------
# Rattachement ASG <-> Target Group (ressource separee en v5+)
# -----------------------------------------------------------------------------
resource "aws_autoscaling_attachment" "app_tg" {
  autoscaling_group_name = aws_autoscaling_group.app.name
  lb_target_group_arn    = aws_lb_target_group.app.arn
}

🟡 Piège health_check_grace_period : si vous mettez < 180s, l'ASG tue l'instance avant que Nextcloud ait fini de démarrer (dnf install + docker pull + container up + DB init, ça prend 2-4 min). 300s est un bon défaut pour ce TP.

Conseils du formateur

Attaquez le certificat self-signed en premier. C'est la ressource la plus "exotique" du module (provider tls que vous ne connaissez pas), et elle bloque le listener HTTPS. Une fois le cert dans ACM, vous êtes tranquille pour le reste de la journée. Beaucoup d'équipes perdent 40 min dessus en fin de matinée parce qu'elles l'ont gardée pour "après" — ne faites pas ça.

Le user_data est le point chaud du débogage. 80 % des "Nextcloud ne démarre pas" viennent d'une variable mal injectée dans le template (une $${variable} écrite ${variable} par erreur, ou l'inverse). Pour débugger, récupérez le user_data rendu avant de lancer l'apply : terraform console → templatefile(...) — si le script rendu compile en bash, vous êtes bons. En cas de souci au boot, SSM Session Manager sur l'EC2 et tail /var/log/user-data.log vous dit tout. Ne cherchez jamais à déboguer en aveugle, l'EC2 vous donnera la réponse en 30 secondes.

Ne touchez pas aux Security Groups ni à l'IAM. Ils sont livrés par le Rôle 5. Si vous avez besoin d'un port ou d'une permission supplémentaire, vous ouvrez un ticket Slack vers le Rôle 5 et il ajoute la règle. Si vous créez un SG "pour aller plus vite", vous violez le contrat d'interface, et le module ne s'intègre plus proprement dans envs/dev. Respectez le périmètre.
