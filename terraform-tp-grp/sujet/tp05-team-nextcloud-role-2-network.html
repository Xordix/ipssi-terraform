🔵 Rappel concept : VPC Endpoint

Un VPC endpoint permet à une instance privée d'atteindre un service AWS (S3, Secrets Manager, etc.) sans passer par Internet (ni NAT, ni IGW). Deux types :

  • Gateway endpoint (S3, DynamoDB) : gratuit, ajoute une route dans la route table concernée. Le trafic part directement vers le service.
  • Interface endpoint (Secrets Manager, KMS, SSM, ...) : payant (~0.01 USD/h + trafic), crée une ENI avec IP privée dans le VPC. Active private_dns_enabled = true pour que secretsmanager.eu-west-1.amazonaws.com résolve vers l'IP privée de l'endpoint.

Bénéfices : réduction des coûts NAT (le trafic S3 et Secrets Manager ne passe plus par la NAT) + sécurité (pas de sortie Internet pour ces flux).

Étapes pas-à-pas (sprint 1)

Étape 1 : Arborescence modules/networking/ (5 min)

Objectif : créer les 6 fichiers du module et poser les versions.

Contexte : un module Terraform bien structuré sépare les préoccupations : variables.tf (entrées), outputs.tf (sorties), locals.tf (calculs internes), main.tf (ressources), versions.tf (contraintes), README.md (doc).

Exemple :

cd tp05-nextcloud/modules/networking

touch main.tf variables.tf outputs.tf locals.tf versions.tf README.md

# Initialiser versions.tf
cat > versions.tf <<'EOF'
terraform {
  required_version = ">= 1.10.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.80"
    }
  }
}
EOF

🔴 Anti-pattern : déclarer un provider "aws" { ... } dans le module. Un module hérite du provider de l'appelant — si vous le déclarez dans le module, vous ne pouvez plus l'utiliser avec un provider aliasé.

🔷 Point clé à vérifier : ls modules/networking/ retourne 6 fichiers. versions.tf contient uniquement terraform { required_version, required_providers }, pas de provider {}.

Étape 3 : locals.tf avec 3 maps AZ → CIDR (15 min)

Objectif : calculer dynamiquement les CIDR des 6 subnets à partir du CIDR VPC.

Contexte : cidrsubnet(prefix, newbits, netnum) découpe un préfixe. Avec vpc_cidr = "10.30.0.0/16", cidrsubnet(var.vpc_cidr, 8, 1) donne "10.30.1.0/24". On définit un offset par type de subnet : public = +1, private_app = +11, private_db = +21. Ça laisse de la marge pour ajouter des subnets plus tard sans chevauchement.

Exemple :

# modules/networking/locals.tf

locals {
  name_prefix = "${var.project_name}-${var.environment}"

  # Subnets publics : /24 a partir de .1.0
  # AZ-a -> 10.30.1.0/24, AZ-b -> 10.30.2.0/24
  public_subnets = {
    for idx, az in var.azs :
    az => cidrsubnet(var.vpc_cidr, 8, idx + 1)
  }

  # Subnets prives app : /24 a partir de .11.0
  # AZ-a -> 10.30.11.0/24, AZ-b -> 10.30.12.0/24
  private_app_subnets = {
    for idx, az in var.azs :
    az => cidrsubnet(var.vpc_cidr, 8, idx + 11)
  }

  # Subnets prives DB : /24 a partir de .21.0
  # AZ-a -> 10.30.21.0/24, AZ-b -> 10.30.22.0/24
  private_db_subnets = {
    for idx, az in var.azs :
    az => cidrsubnet(var.vpc_cidr, 8, idx + 21)
  }
}

Résultat attendu : avec vpc_cidr = "10.30.0.0/16" et azs = ["eu-west-1a", "eu-west-1b"], les 3 maps contiennent :

public_subnets      = { eu-west-1a = "10.30.1.0/24",  eu-west-1b = "10.30.2.0/24"  }
private_app_subnets = { eu-west-1a = "10.30.11.0/24", eu-west-1b = "10.30.12.0/24" }
private_db_subnets  = { eu-west-1a = "10.30.21.0/24", eu-west-1b = "10.30.22.0/24" }

À vous !

Testez vos CIDR en console : terraform console dans envs/dev/ puis cidrsubnet("10.30.0.0/16", 8, 11) — doit retourner "10.30.11.0/24". Si vous obtenez autre chose, vous avez un bug d'offset.

Indices :

  • Le 2e argument de cidrsubnet (newbits) est le nombre de bits à ajouter au préfixe. /16 + 8 = /24.
  • Le 3e argument (netnum) est l'index du sous-réseau. Commence à 0.

🔷 Point clé à vérifier : les 3 maps n'ont aucun CIDR qui se chevauche. Vérification mentale : .1.0, .2.0, .11.0, .12.0, .21.0, .22.0 — OK.

Étape 4 : main.tf — VPC + IGW (10 min)

Objectif : créer le VPC et l'Internet Gateway.

Contexte : enable_dns_support et enable_dns_hostnames doivent être à true pour que les VPC endpoints avec DNS privé fonctionnent. Sans ça, l'endpoint Secrets Manager ne résoudra pas son nom DNS depuis l'EC2.

Exemple :

# modules/networking/main.tf

# =============================================================================
# VPC
# =============================================================================
resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr
  enable_dns_support   = true  # requis pour les VPCE avec DNS prive
  enable_dns_hostnames = true  # idem

  tags = {
    Name = "${local.name_prefix}-vpc"
  }
}

# =============================================================================
# Internet Gateway - seule porte de sortie des subnets publics
# =============================================================================
resource "aws_internet_gateway" "main" {
  vpc_id = aws_vpc.main.id

  tags = {
    Name = "${local.name_prefix}-igw"
  }
}

🔷 Point clé à vérifier : dans la console AWS VPC, le VPC apparaît avec les deux DNS flags à "Enabled". Si vous oubliez enable_dns_hostnames, les endpoints cassent en aval — le piège le plus classique.

Étape 5 : Les 6 subnets (3 ressources for_each) (15 min)

Objectif : créer les 6 subnets via 3 ressources distinctes (une par type), chacune avec for_each sur la map correspondante.

Contexte : 3 ressources distinctes et non une seule avec tous les subnets — ça rend les outputs propres (public_subnet_ids vs private_db_subnet_ids) et évite les erreurs de tri par type. Le for_each sur une map utilise la clé (AZ) pour garantir la stabilité des adresses Terraform.

Exemple :

# =============================================================================
# Subnets publics (1 par AZ)
# =============================================================================
resource "aws_subnet" "public" {
  for_each = local.public_subnets

  vpc_id                  = aws_vpc.main.id
  cidr_block              = each.value
  availability_zone       = each.key
  map_public_ip_on_launch = true  # IP publique auto pour EC2 lance ici (rarement utile)

  tags = {
    Name = "${local.name_prefix}-public-${each.key}"
    Tier = "public"
  }
}

# =============================================================================
# Subnets prives app (ASG + EC2 Nextcloud)
# =============================================================================
resource "aws_subnet" "private_app" {
  for_each = local.private_app_subnets

  vpc_id            = aws_vpc.main.id
  cidr_block        = each.value
  availability_zone = each.key

  tags = {
    Name = "${local.name_prefix}-private-app-${each.key}"
    Tier = "private-app"
  }
}

# =============================================================================
# Subnets prives DB (RDS)
# =============================================================================
resource "aws_subnet" "private_db" {
  for_each = local.private_db_subnets

  vpc_id            = aws_vpc.main.id
  cidr_block        = each.value
  availability_zone = each.key

  tags = {
    Name = "${local.name_prefix}-private-db-${each.key}"
    Tier = "private-db"
  }
}

À vous !

Vérifiez que terraform plan dans envs/dev/ affiche exactement 6 aws_subnet à créer : aws_subnet.public["eu-west-1a"], aws_subnet.public["eu-west-1b"], etc.

Indices :

  • Le for_each itère sur la map local.public_subnets dont les clés sont les AZ.
  • each.key = AZ (string), each.value = CIDR (string).

🔷 Point clé à vérifier : terraform state list | grep subnet affichera 6 lignes après apply — 2 par type.

Étape 6 : EIP + NAT Gateway (10 min)

Objectif : créer une EIP et une NAT Gateway single dans le subnet public AZ-a.

Contexte : la NAT Gateway coûte ~1 USD/jour + trafic. Pour ce TP, une seule NAT (et pas une par AZ comme en prod) — c'est un compromis budget/résilience acceptable pour un env de dev.

Exemple :

# =============================================================================
# Elastic IP pour la NAT Gateway
# =============================================================================
resource "aws_eip" "nat" {
  domain = "vpc"  # requis depuis 2022 (l ancien 'vpc = true' est deprecated)

  tags = {
    Name = "${local.name_prefix}-nat-eip"
  }

  # Il faut l IGW pour allouer une EIP utilisable en VPC
  depends_on = [aws_internet_gateway.main]
}

# =============================================================================
# NAT Gateway (single AZ pour economie)
# =============================================================================
resource "aws_nat_gateway" "main" {
  allocation_id = aws_eip.nat.id
  # On place la NAT dans le subnet public de la PREMIERE AZ
  subnet_id = aws_subnet.public[var.azs[0]].id

  tags = {
    Name = "${local.name_prefix}-nat"
  }

  depends_on = [aws_internet_gateway.main]
}

🟡 Piège courant : oublier depends_on = [aws_internet_gateway.main] sur l'EIP. Terraform essaie d'allouer l'EIP avant que l'IGW soit prêt → InvalidAllocationID.NotFound au plan suivant.

🔷 Point clé à vérifier : après apply, aws ec2 describe-nat-gateways --region eu-west-1 retourne 1 NAT en état available, dans l'AZ-a.

Étape 7 : Route tables + 6 associations (15 min)

Objectif : créer 2 route tables (publique vers IGW, privée vers NAT) et associer les 6 subnets.

Contexte : les subnets DB utilisent la même route table que les subnets app (la NAT) — c'est OK, RDS n'a pas besoin de NAT en fonctionnement normal, mais ça permet à RDS d'atteindre des services AWS (ex: pour les snapshots KMS) si besoin.

Exemple :

# =============================================================================
# Route Table publique : 0.0.0.0/0 -> IGW
# =============================================================================
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id

  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.main.id
  }

  tags = {
    Name = "${local.name_prefix}-public-rt"
  }
}

# =============================================================================
# Route Table privee : 0.0.0.0/0 -> NAT Gateway
# =============================================================================
resource "aws_route_table" "private" {
  vpc_id = aws_vpc.main.id

  route {
    cidr_block     = "0.0.0.0/0"
    nat_gateway_id = aws_nat_gateway.main.id
  }

  tags = {
    Name = "${local.name_prefix}-private-rt"
  }
}

# =============================================================================
# Associations : 2 publics -> RT publique
# =============================================================================
resource "aws_route_table_association" "public" {
  for_each = aws_subnet.public

  subnet_id      = each.value.id
  route_table_id = aws_route_table.public.id
}

# =============================================================================
# Associations : 2 private_app -> RT privee
# =============================================================================
resource "aws_route_table_association" "private_app" {
  for_each = aws_subnet.private_app

  subnet_id      = each.value.id
  route_table_id = aws_route_table.private.id
}

# =============================================================================
# Associations : 2 private_db -> RT privee
# =============================================================================
resource "aws_route_table_association" "private_db" {
  for_each = aws_subnet.private_db

  subnet_id      = each.value.id
  route_table_id = aws_route_table.private.id
}

Vérifiez avec terraform plan qu'il y a bien 2 aws_route_table + 6 aws_route_table_association à créer.

Étape 8 : VPC Endpoint S3 (gateway) (10 min)

Objectif : créer un endpoint gateway S3 attaché à la route table privée.

🔵 Rappel concept : Gateway endpoint S3

Un gateway endpoint est gratuit (contrairement aux interface endpoints). Il injecte une route spéciale (pour les préfixes IP régionaux S3) dans la route table attachée. Résultat : tout appel à S3 depuis un subnet privé sort par l'endpoint et non par le NAT → zéro coût de sortie S3, et le trafic ne passe jamais par Internet public.

Exemple :

# =============================================================================
# VPC Endpoint S3 - gateway, gratuit
# =============================================================================
resource "aws_vpc_endpoint" "s3" {
  vpc_id            = aws_vpc.main.id
  service_name      = "com.amazonaws.eu-west-1.s3"
  vpc_endpoint_type = "Gateway"

  # Injecter la route dans la RT privee
  # (les subnets publics peuvent utiliser l IGW directement pour S3)
  route_table_ids = [aws_route_table.private.id]

  tags = {
    Name = "${local.name_prefix}-vpce-s3"
  }
}

🔷 Point clé à vérifier : dans la console VPC → Endpoints, l'endpoint com.amazonaws.eu-west-1.s3 est en état Available, type Gateway, avec la route table privée associée.

Étape 9 : VPC Endpoint Secrets Manager (interface) + SG (15 min)

Objectif : créer un interface endpoint pour Secrets Manager + un SG "endpoints" qui autorise 443 depuis le VPC CIDR.

🔵 Rappel concept : Interface endpoint

Contrairement au gateway endpoint, un interface endpoint est payant (~0.01 USD/h + trafic) mais indispensable pour les services qui ne supportent pas le gateway (Secrets Manager, KMS, SSM, etc.). Il crée une ENI (interface réseau) avec une IP privée dans chaque subnet listé. Le private_dns_enabled = true fait que secretsmanager.eu-west-1.amazonaws.com résout vers l'IP privée de l'ENI — le code AWS SDK fonctionne sans modification.

Un SG dédié est attaché à l'ENI : il doit autoriser 443 depuis le VPC CIDR.

🔷 Décision prise au kick-off : le SG "endpoints" est créé par le Rôle 2 (vous), pas par le Rôle 5. Ça évite un aller-retour : vous exposez vpc_endpoints_security_group_id en output, personne n'en a besoin côté Rôle 5.

Exemple :

# =============================================================================
# SG dedie aux VPC endpoints : autorise 443 depuis le VPC CIDR
# =============================================================================
resource "aws_security_group" "vpc_endpoints" {
  name        = "${local.name_prefix}-vpce-sg"
  description = "Autorise HTTPS depuis VPC vers les VPC endpoints"
  vpc_id      = aws_vpc.main.id

  tags = {
    Name = "${local.name_prefix}-vpce-sg"
  }
}

resource "aws_vpc_security_group_ingress_rule" "vpce_https_from_vpc" {
  security_group_id = aws_security_group.vpc_endpoints.id

  description = "HTTPS 443 depuis le VPC"
  from_port   = 443
  to_port     = 443
  ip_protocol = "tcp"
  cidr_ipv4   = var.vpc_cidr
}

# =============================================================================
# VPC Endpoint Secrets Manager - interface, DNS prive
# =============================================================================
resource "aws_vpc_endpoint" "secretsmanager" {
  vpc_id              = aws_vpc.main.id
  service_name        = "com.amazonaws.eu-west-1.secretsmanager"
  vpc_endpoint_type   = "Interface"
  private_dns_enabled = true

  # Deployer le endpoint dans les 2 subnets prives app (HA entre AZ)
  subnet_ids = [for s in aws_subnet.private_app : s.id]

  security_group_ids = [aws_security_group.vpc_endpoints.id]

  tags = {
    Name = "${local.name_prefix}-vpce-secretsmanager"
  }
}

🟡 Piège : si vous oubliez private_dns_enabled = true, l'EC2 ne peut pas utiliser le nom DNS standard Secrets Manager — elle devra utiliser le DNS de l'endpoint directement (URL très longue). Le user_data du Rôle 3 suppose le DNS standard.

🟠 Alerte coût : un interface endpoint coûte ~7 €/mois par endpoint par AZ. 2 subnets = 2 ENI = 2x le coût. Acceptable pour un TP d'une journée, à supprimer en fin de journée avec destroy.

À vous !

Ajoutez ces ressources à votre main.tf. Si vous avez le temps (étape 10 bonus), ajoutez un 3e endpoint pour S3 en mode interface (utile pour les PUT/GET scoped par policy) — mais attention, ça double les coûts d'endpoints.

Indices :

  • service_name pour Secrets Manager : com.amazonaws.<region>.secretsmanager
  • subnet_ids attend une liste, pas une map → utiliser [for s in aws_subnet.private_app : s.id]

🔷 Point clé à vérifier : aws ec2 describe-vpc-endpoints --region eu-west-1 retourne 2 endpoints en Available. Test depuis l'EC2 (quand Rôle 3 sera livré) : nslookup secretsmanager.eu-west-1.amazonaws.com doit retourner une IP privée 10.30.x.y.

Conseils du formateur

Pièges typiques du Network Engineer — trois choses qui font perdre des points. D'abord, ne pas respecter le type map des outputs subnets. Si vous exposez output "public_subnet_ids" { value = [for s in aws_subnet.public : s.id] } (liste au lieu de map), le Rôle 3 va vous détester car son for_each sur l'ALB casse. Respectez { for k, s in aws_subnet.public : k => s.id }. Ensuite, oublier enable_dns_hostnames = true sur le VPC — le symptôme n'apparaît qu'au TP du Rôle 3 (user_data qui n'arrive pas à résoudre Secrets Manager), et à ce moment-là vous êtes en sprint 3, le formateur passe à autre chose. Enfin, placer la NAT Gateway dans un subnet privé — erreur classique : une NAT doit être dans un subnet public (pour pouvoir atteindre l'IGW). Relisez l'étape 6.

Ordre recommandé — Étapes 1 → 2 → 3 → 4 → 5 → 6 → 7 dans l'ordre listé. Entre 4 et 5, lancez un premier terraform validate dans envs/dev/ (il va râler sur les modules manquants mais au moins votre module networking est validé). Les étapes 8 et 9 (VPC endpoints) peuvent être sautées en cas de retard — le TP fonctionne sans, au prix d'un peu plus de trafic NAT. Si vous êtes à l'heure, faites les deux. L'étape 10 (README + outputs) doit absolument être livrée avant la pause déjeuner, c'est le contrat avec les autres rôles.

Quand demander de l'aide — Si après 30 minutes sur un cidrsubnet() qui ne retourne pas la bonne valeur, vous êtes bloqué. Ouvrez terraform console et testez interactivement : cidrsubnet("10.30.0.0/16", 8, 1) → 10.30.1.0/24. Si vous êtes bloqué sur un VPC endpoint qui reste en Pending plus de 5 minutes, c'est souvent un SG mal configuré (port 443 pas ouvert depuis le VPC CIDR) — inutile d'attendre, relisez l'étape 9. Et surtout : ne modifiez pas la NAT Gateway ou l'EIP via la console pour "tester" — le drift va vous faire perdre 20 minutes à réaligner le state.
