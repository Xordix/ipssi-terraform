🔵 Rappel concept : backend S3

Par défaut, Terraform stocke son terraform.tfstate localement dans le dossier courant. Pour travailler en équipe, on met ce state dans un bucket S3 avec verrou (lock) : impossible d'avoir 2 apply simultanés. Depuis Terraform 1.10, le lock est natif via use_lockfile = true — plus besoin de DynamoDB.

terraform {
  backend "s3" {
    bucket       = "tf-state-kolab"
    key          = "envs/dev/terraform.tfstate"
    region       = "eu-west-1"
    encrypt      = true
    use_lockfile = true
  }
}

🔵 Rappel concept : bootstrap "chicken-and-egg"

Le bucket qui stocke le state est lui-même de l'infra AWS. On ne peut pas le créer via Terraform (où stockerait-on son state ?). Solution : script bash AWS CLI one-shot exécuté avant le premier terraform init. C'est l'objet de bootstrap/create-state-bucket.sh.

🔵 Rappel concept : organisation modules/ + envs/

Un module Terraform = un dossier .tf avec des variable (inputs) + output (sorties). L'appelant passe des valeurs via module "x" { source = "../../modules/x" ... } et lit les résultats via module.x.<output_name>. Ici on sépare les modules réutilisables (dans modules/) de l'environnement qui les orchestre (dans envs/dev/).

module "networking" {
  source       = "../../modules/networking"
  project_name = "kolab"
  vpc_cidr     = "10.30.0.0/16"
}

module "compute" {
  source = "../../modules/compute"
  vpc_id = module.networking.vpc_id   # consomme l'output
}

Étape 2 : Écrire le script bootstrap bucket state + KMS CMK (25 min)

Objectif : créer le bucket S3 qui va stocker les states + une CMK KMS qui chiffre ce bucket. Opération one-shot hors Terraform (problème de "chicken and egg" — Terraform a besoin d'un backend pour s'initialiser, donc le backend doit exister avant le premier init).

🔵 Rappel concept : KMS Customer Managed Key (CMK)

AWS KMS gère les clés de chiffrement. Une CMK (Customer Managed Key) est une clé dont vous gérez la policy (à la différence des clés AWS-managed comme aws/s3). Ici on crée une CMK dédiée au state Terraform, avec rotation annuelle activée. L'alias (alias/tf-state-xxx) est un nom humain plus simple que l'ID UUID.

Contexte : contrairement à TP01 qui utilisait SSE-S3 (AES256 managé par AWS), ici on impose KMS CMK avec rotation automatique — exigence T5 du CdC. La CMK doit être créée avant le bucket pour pouvoir chiffrer le bucket à sa création.

Exemple :

#!/usr/bin/env bash
# bootstrap/create-state-bucket.sh
# One-shot : cree la KMS CMK puis le bucket S3 de state.

set -euo pipefail

USERNAME="${USERNAME:-kolab-team1}"
REGION="${REGION:-eu-west-1}"

BUCKET="tf-state-${USERNAME}-kolab"
KMS_ALIAS="alias/tf-state-${USERNAME}"

echo "=== 1. Creation KMS CMK pour chiffrer le state ==="

# Creer la CMK si elle n existe pas deja
KMS_KEY_ID=$(aws kms list-aliases --region "${REGION}" \
  --query "Aliases[?AliasName=='${KMS_ALIAS}'].TargetKeyId" --output text)

if [ -z "${KMS_KEY_ID}" ]; then
  KMS_KEY_ID=$(aws kms create-key \
    --region "${REGION}" \
    --description "CMK chiffrement du state Terraform Kolab" \
    --key-usage ENCRYPT_DECRYPT \
    --key-spec SYMMETRIC_DEFAULT \
    --query 'KeyMetadata.KeyId' --output text)

  # Activer la rotation annuelle (exigence securite)
  aws kms enable-key-rotation --region "${REGION}" --key-id "${KMS_KEY_ID}"

  # Creer l alias pour s y referer facilement
  aws kms create-alias --region "${REGION}" \
    --alias-name "${KMS_ALIAS}" \
    --target-key-id "${KMS_KEY_ID}"
else
  echo "CMK ${KMS_ALIAS} existe deja."
fi

KMS_KEY_ARN=$(aws kms describe-key --region "${REGION}" \
  --key-id "${KMS_KEY_ID}" --query 'KeyMetadata.Arn' --output text)

echo "KMS CMK ARN : ${KMS_KEY_ARN}"

echo ""
echo "=== 2. Creation bucket S3 state ==="

if aws s3api head-bucket --bucket "${BUCKET}" 2>/dev/null; then
  echo "Bucket ${BUCKET} existe deja."
else
  aws s3api create-bucket \
    --bucket "${BUCKET}" \
    --region "${REGION}" \
    --create-bucket-configuration "LocationConstraint=${REGION}"
fi

# Versioning - indispensable pour recuperer un state corrompu
aws s3api put-bucket-versioning \
  --bucket "${BUCKET}" \
  --versioning-configuration Status=Enabled

# Chiffrement KMS CMK (et plus SSE-S3)
aws s3api put-bucket-encryption \
  --bucket "${BUCKET}" \
  --server-side-encryption-configuration "{
    \"Rules\": [{
      \"ApplyServerSideEncryptionByDefault\": {
        \"SSEAlgorithm\": \"aws:kms\",
        \"KMSMasterKeyID\": \"${KMS_KEY_ARN}\"
      },
      \"BucketKeyEnabled\": true
    }]
  }"

# Block Public Access complet
aws s3api put-public-access-block \
  --bucket "${BUCKET}" \
  --public-access-block-configuration \
    "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"

# Bucket policy : refuse tout acces non-HTTPS
aws s3api put-bucket-policy \
  --bucket "${BUCKET}" \
  --policy "$(cat <<EOF
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "DenyInsecureTransport",
    "Effect": "Deny",
    "Principal": "*",
    "Action": "s3:*",
    "Resource": [
      "arn:aws:s3:::${BUCKET}",
      "arn:aws:s3:::${BUCKET}/*"
    ],
    "Condition": { "Bool": { "aws:SecureTransport": "false" } }
  }]
}
EOF
)"

echo ""
echo "==============================================="
echo " Bootstrap OK."
echo "   Bucket : ${BUCKET}"
echo "   KMS    : ${KMS_ALIAS} (${KMS_KEY_ARN})"
echo ""
echo " Adapter envs/dev/backend.tf avec ces valeurs."
echo "==============================================="

Étape 3 : envs/dev/backend.tf + providers.tf (15 min)

Objectif : pointer le state sur le bucket créé à l'étape 2 et déclarer le provider AWS avec des tags par défaut.

🔵 Rappel concept : default_tags du provider AWS

Le bloc default_tags dans provider "aws" {} applique automatiquement un jeu de tags à toutes les ressources AWS du state — indispensable pour le chargeback et le FinOps. Plus besoin de re-taguer chaque ressource individuellement.

Contexte : depuis Terraform 1.10, le backend S3 supporte le locking natif via use_lockfile = true (plus besoin de DynamoDB).

Exemple :

# =============================================================================
# envs/dev/backend.tf
# Backend S3 natif avec locking. NE PAS committer le tfstate.
# =============================================================================

terraform {
  required_version = ">= 1.10.0"

  backend "s3" {
    bucket       = "tf-state-kolab-team1-kolab"  # <- adapter au bucket cree
    key          = "envs/dev/terraform.tfstate"
    region       = "eu-west-1"
    encrypt      = true
    kms_key_id   = "alias/tf-state-kolab-team1"  # <- adapter a votre alias
    use_lockfile = true                          # TF >= 1.10 : locking natif
  }
}

# =============================================================================
# envs/dev/providers.tf
# Provider AWS avec default_tags pour taggage transversal.
# =============================================================================

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.80"
    }
  }
}

provider "aws" {
  region = var.aws_region

  default_tags {
    tags = {
      Project     = var.project_name
      Environment = var.environment
      ManagedBy   = "Terraform"
      Team        = "TP05-equipe"
    }
  }
}

À vous !

Créez ces deux fichiers en adaptant le nom du bucket et l'alias KMS à ceux que votre script bootstrap a créés. Ne committez jamais de vraies valeurs secrètes — ici ce sont des noms publics, donc c'est OK.

Indices :

  • use_lockfile = true remplace dynamodb_table (TF ≥ 1.10)
  • kms_key_id accepte un ARN ou un alias

Étape 4 : envs/dev/variables.tf + terraform.tfvars.example (10 min)

Objectif : exposer les deux seules variables configurables de l'environnement : région + CIDR d'admin.

Contexte : un envs/<env> doit avoir le strict minimum de variables. Tout le reste (CIDR VPC, AZs, type d'instance) est dans les modules ou dans locals. Moins il y a de variables, moins il y a de risque de dérive entre dev et staging.

Exemple :

# =============================================================================
# envs/dev/variables.tf
# =============================================================================

variable "aws_region" {
  description = "Region AWS (impose : eu-west-1 pour RGPD)."
  type        = string
  default     = "eu-west-1"

  validation {
    condition     = var.aws_region == "eu-west-1"
    error_message = "La region doit etre eu-west-1 (exigence T1 du CdC)."
  }
}

variable "allowed_admin_cidr" {
  description = "CIDR IP autorise a atteindre l ALB en HTTPS (IP formateur)."
  type        = string

  validation {
    condition     = can(cidrhost(var.allowed_admin_cidr, 0))
    error_message = "Doit etre un CIDR IPv4 valide (ex: 203.0.113.42/32)."
  }
}

variable "project_name" {
  description = "Nom de projet pour le tagging."
  type        = string
  default     = "kolab"
}

variable "environment" {
  description = "Nom de l environnement."
  type        = string
  default     = "dev"
}

# envs/dev/terraform.tfvars.example
# Copier en terraform.tfvars (gitignored) et renseigner.

allowed_admin_cidr = "203.0.113.42/32"

À vous !

Ajoutez une validation sur allowed_admin_cidr qui refuse 0.0.0.0/0 (ouvrir au monde entier serait un anti-pattern pour un TP).

Indices :

  • condition = var.allowed_admin_cidr != "0.0.0.0/0"

Étape 5 : envs/dev/main.tf — les 4 blocks module (30 min)

Objectif : câbler les 4 modules entre eux en respectant les interfaces figées au kick-off.

Contexte : c'est le fichier qui matérialise l'architecture. Il doit être écrit avant que les autres rôles codent leur module — il sert de contrat. Chaque rôle a sa liste d'input à remplir et d'output à exposer. Tant que les source = "../../modules/..." n'existent pas, Terraform refuse de init — c'est normal, on débloque l'équipe en livrant ce fichier en premier.

Exemple :

# =============================================================================
# envs/dev/main.tf
# Orchestration des 4 modules. Interfaces FIGEES au kick-off.
# =============================================================================

# -----------------------------------------------------------------------------
# Role 2 : reseau (VPC, subnets, NAT, endpoints)
# -----------------------------------------------------------------------------
module "networking" {
  source = "../../modules/networking"

  project_name = var.project_name
  environment  = var.environment
  vpc_cidr     = "10.30.0.0/16"
  azs          = ["eu-west-1a", "eu-west-1b"]
}

# -----------------------------------------------------------------------------
# Role 5 : securite (SGs, KMS, IAM, Secrets Manager)
# Note : consomme les ARN S3 de data (late binding).
# -----------------------------------------------------------------------------
module "security" {
  source = "../../modules/security"

  project_name          = var.project_name
  environment           = var.environment
  vpc_id                = module.networking.vpc_id
  vpc_cidr              = module.networking.vpc_cidr
  allowed_admin_cidr    = var.allowed_admin_cidr
  s3_primary_bucket_arn = module.data.s3_primary_bucket_arn
  s3_logs_bucket_arn    = module.data.s3_logs_bucket_arn
}

# -----------------------------------------------------------------------------
# Role 4 : data (RDS PG Multi-AZ + 2 buckets S3 chiffres KMS)
# -----------------------------------------------------------------------------
module "data" {
  source = "../../modules/data"

  project_name           = var.project_name
  environment            = var.environment
  vpc_id                 = module.networking.vpc_id
  private_db_subnet_ids  = module.networking.private_db_subnet_ids
  db_security_group_id   = module.security.db_security_group_id
  kms_key_arn            = module.security.kms_key_arn
  db_password_secret_arn = module.security.db_password_secret_arn
}

# -----------------------------------------------------------------------------
# Role 3 : compute (ALB, ASG, EC2, user_data Nextcloud)
# -----------------------------------------------------------------------------
module "compute" {
  source = "../../modules/compute"

  project_name                 = var.project_name
  environment                  = var.environment
  vpc_id                       = module.networking.vpc_id
  public_subnet_ids            = module.networking.public_subnet_ids
  private_app_subnet_ids       = module.networking.private_app_subnet_ids
  alb_security_group_id        = module.security.alb_security_group_id
  app_security_group_id        = module.security.app_security_group_id
  app_instance_profile_name    = module.security.app_instance_profile_name
  db_endpoint                  = module.data.db_endpoint
  db_name                      = module.data.db_name
  db_username                  = module.data.db_username
  db_password_secret_arn       = module.security.db_password_secret_arn
  admin_password_secret_arn    = module.security.admin_password_secret_arn
  s3_primary_bucket_name       = module.data.s3_primary_bucket_name
  s3_logs_bucket_name          = module.data.s3_logs_bucket_name
}

Étape 8 : global/iam-github-oidc/ (bonus — 30 min)

Objectif : permettre à un workflow GitHub Actions d'assumer un rôle IAM AWS sans aucun secret stocké dans GitHub (ni clé d'accès, ni mot de passe). Le token OIDC émis par GitHub est échangé contre des credentials AWS temporaires (1h).

Contexte : dans une vraie équipe Ops, c'est la seule manière correcte d'avoir de la CI/CD AWS. Les aws-access-key-id dans les secrets GitHub sont un anti-pattern (rotation manuelle, risque de leak).

Exemple :

# global/iam-github-oidc/main.tf

# Provider OIDC GitHub - un seul par compte AWS
resource "aws_iam_openid_connect_provider" "github" {
  url             = "https://token.actions.githubusercontent.com"
  client_id_list  = ["sts.amazonaws.com"]
  thumbprint_list = ["6938fd4d98bab03faadb97b34396831e3780aea1"]

  tags = {
    Name = "github-actions-oidc"
  }
}

# Role assumable par le workflow du repo <org>/<repo>
resource "aws_iam_role" "github_actions" {
  name = "kolab-github-actions-tf-plan"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect = "Allow"
      Principal = {
        Federated = aws_iam_openid_connect_provider.github.arn
      }
      Action = "sts:AssumeRoleWithWebIdentity"
      Condition = {
        StringEquals = {
          "token.actions.githubusercontent.com:aud" = "sts.amazonaws.com"
        }
        StringLike = {
          # IMPORTANT : scoper le trust au repo pour eviter l abus
          "token.actions.githubusercontent.com:sub" = "repo:${var.github_org}/${var.github_repo}:*"
        }
      }
    }]
  })
}

# Policy en lecture pour plan. Pour apply, creer un role separe avec droits etendus.
resource "aws_iam_role_policy_attachment" "read_only" {
  role       = aws_iam_role.github_actions.name
  policy_arn = "arn:aws:iam::aws:policy/ReadOnlyAccess"
}

À vous !

Ajoutez variables.tf, outputs.tf (expose role_arn), et backend.tf avec key = "global/iam-github-oidc/terraform.tfstate". Appliquez ce module avant d'écrire le workflow — sinon le workflow échoue avec could not assume role.

🔷 Point clé à vérifier : terraform output role_arn dans global/iam-github-oidc/ retourne un ARN de la forme arn:aws:iam::123456789012:role/kolab-github-actions-tf-plan.

Étape 9 : .github/workflows/terraform-plan.yml (bonus — 20 min)

Objectif : déclencher un terraform plan automatique sur chaque PR vers main, avec commentaire du diff sur la PR.

Contexte : c'est le minimum syndical d'une CI/CD Terraform en entreprise. L'apply automatique sur merge est possible mais hors scope de ce TP (risque trop élevé sur un TP).

Exemple :

# .github/workflows/terraform-plan.yml

name: Terraform Plan

on:
  pull_request:
    branches: [main]
    paths:

      - 'envs/dev/**'
      - 'modules/**'

permissions:
  id-token: write     # indispensable pour OIDC
  contents: read
  pull-requests: write

jobs:
  plan:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: envs/dev

    steps:

      - uses: actions/checkout@v4

      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/kolab-github-actions-tf-plan
          aws-region: eu-west-1

      - uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: 1.10.3

      - name: Terraform fmt
        run: terraform fmt -check -recursive

      - name: Terraform init
        run: terraform init -input=false

      - name: Terraform validate
        run: terraform validate

      - name: Terraform plan
        id: plan
        run: terraform plan -no-color -input=false -out=tfplan
        continue-on-error: true

      - name: Comment PR
        uses: actions/github-script@v7
        with:
          script: |
            const output = `### Terraform Plan

\`\`\`
${{ steps.plan.outputs.stdout }}
\`\`\`
`;
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: output
            });

À vous !

Adaptez role-to-assume avec l'ARN exporté par global/iam-github-oidc/. Ouvrez une PR vide pour tester — le workflow doit tourner en < 2 minutes.

🔷 Point clé à vérifier : sur l'onglet Actions de GitHub, le job plan est vert, et un commentaire est posté sur la PR avec le résumé du plan.

Conseils du formateur

Pièges typiques du Platform Lead — 3 choses qui font perdre du temps sur ce rôle. D'abord, la tentation de commencer par le code métier. Ne le faites pas. Votre valeur ajoutée est de livrer le bootstrap + l'arborescence + les interfaces figées dans l'heure qui suit le kick-off. Si vous commencez à coder un module, vous bloquez les 4 autres rôles. Ensuite, la modification tardive d'une interface. Si au milieu du sprint un rôle vous demande de renommer un output, refusez et proposez une PR séparée post-TP. Les interfaces figées, c'est figé. Enfin, ne pas tester le bootstrap sur votre propre compte AWS avant de le distribuer. Si votre script plante sur le compte du formateur, vous perdez 30 minutes que vous ne rattraperez pas.

Ordre recommandé — Étapes 1 → 2 → 3 → 5 → 4 → 6 → 7 pendant le sprint 1 (2h30), puis sprint 2 en revue de PR (1h). L'étape 5 (main.tf avec les 4 blocks module) avant les variables (étape 4) car c'est le fichier qui débloque les autres rôles — vous pouvez renseigner le nom des variables plus tard. Les étapes 8-9 (OIDC + CI) uniquement si vous avez terminé tout le reste à 14h, sinon vous risquez de ne pas finir les étapes critiques. Le packaging zip final se fait en toute fin de journée, pas avant — sinon vous zippez un state qui pointe sur des ressources vivantes.

Quand demander de l'aide — Vous avez droit à 10 minutes de support formateur. Gardez-les pour un cas vraiment bloquant : AccessDenied KMS que vous ne comprenez pas, ou trust policy OIDC qui refuse le token. Pas pour un terraform fmt qui ne passe pas — ça, vous le débuggez seul en lisant le message. Si un rôle est en retard et menace votre gate de fin de journée, escaladez tôt (avant 14h) au formateur pour qu'il aide ce rôle en priorité.
