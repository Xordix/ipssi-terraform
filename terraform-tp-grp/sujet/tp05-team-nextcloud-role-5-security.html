🔵 Rappel concept : KMS CMK + key policy

AWS KMS gère les clés de chiffrement. Une CMK (Customer Managed Key) = clé dont vous gérez la policy (à la différence des clés AWS-managed type aws/s3). Elle chiffre ici S3, RDS et Secrets Manager.

🟥 Règle d'or : la key policy doit toujours donner kms:* au compte root. Sinon la clé est brickée — aucun moyen de la récupérer, il faut attendre deletion_window_in_days (7 à 30 j).

resource "aws_kms_key" "main" {
  enable_key_rotation = true
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Sid       = "EnableRootAccountAccess"
      Effect    = "Allow"
      Principal = { AWS = "arn:aws:iam::${data.aws_caller_identity.current.account_id}:root" }
      Action    = "kms:*"
      Resource  = "*"
    }]
  })
}

🔵 Rappel concept : Secrets Manager

Service AWS qui stocke des secrets (mots de passe, tokens) chiffrés avec KMS, avec versioning et accès contrôlé par IAM. Ici on génère le mot de passe avec random_password et on le stocke dans un aws_secretsmanager_secret chiffré avec votre CMK.

resource "random_password" "db" { length = 24, special = false }

resource "aws_secretsmanager_secret" "db_password" {
  name       = "kolab-dev-db-password"
  kms_key_id = aws_kms_key.main.arn
}

resource "aws_secretsmanager_secret_version" "db_password" {
  secret_id     = aws_secretsmanager_secret.db_password.id
  secret_string = random_password.db.result
}

🔵 Rappel concept : IAM role + Instance Profile

Un IAM role est un ensemble de permissions qui peut être "endossé" par un principal (service, user, autre role). Pour qu'une EC2 porte un role, on l'enrobe dans un instance profile :

  • L'assume_role_policy autorise le service ec2.amazonaws.com à endosser le rôle
  • Des aws_iam_role_policy y sont attachées (S3, Secrets, KMS scopées aux ARNs précis)
  • L'aws_iam_instance_profile wrap le rôle et est consommé par le launch template du Rôle 3

Résultat : l'EC2 accède à AWS sans aucune clé en dur — le SDK utilise l'IMDS pour récupérer des credentials éphémères.

resource "aws_iam_role" "app" {
  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = { Service = "ec2.amazonaws.com" }
      Action    = "sts:AssumeRole"
    }]
  })
}

resource "aws_iam_instance_profile" "app" {
  name = "nextcloud-app-profile"
  role = aws_iam_role.app.name
}

Étape 3 : KMS CMK + alias (25 min)

Objectif : créer la clé KMS customer-managed qui chiffrera S3, RDS et Secrets Manager.

Contexte :

La KMS CMK est la pièce maîtresse. Si la key policy est mal écrite :

  • Le root account est bloqué → la clé est brickée (aucun moyen de la récupérer, il faut attendre le deletion_window).
  • Le rôle app ne peut pas déchiffrer les secrets → Nextcloud spinne à l'infini au boot.
  • RDS ne peut pas chiffrer son storage → l'instance RDS échoue à démarrer.

🟥 Règle d'or : toujours donner tous les droits au root account dans la key policy. C'est le seul moyen de récupérer la main si on se trompe ailleurs.

Explication — pattern de key policy :

flowchart LR
    t_root[Root account
kms:*]
    t_app[Role app runtime
kms:Decrypt
kms:DescribeKey]
    t_services[Services AWS
s3 rds secretsmanager
via IAM policies]

    t_policy{Key policy}

    t_root --> t_policy
    t_app --> t_policy
    t_services -.via grants.-> t_policy

    t_policy --> t_key[(KMS CMK)]

    classDef root fill:#fff3cd,stroke:#ffc107
    classDef app fill:#d1e7dd,stroke:#198754
    classDef svc fill:#cfe2ff,stroke:#0d6efd
    class t_root root
    class t_app app
    class t_services svc

Exemple :

# kms.tf
# CMK principale : chiffre S3, RDS, Secrets Manager pour cet environnement.
resource "aws_kms_key" "main" {
  description             = "CMK principale ${local.name_prefix} - chiffrement S3 RDS Secrets"
  deletion_window_in_days = 10
  enable_key_rotation     = true   # Rotation annuelle automatique (obligatoire en prod)

  # Key policy : qui peut faire quoi avec cette cle.
  # IMPORTANT : le root account doit avoir kms:* sinon la cle est brickee.
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid    = "EnableRootAccountAccess"
        Effect = "Allow"
        Principal = {
          AWS = "arn:${data.aws_partition.current.partition}:iam::${data.aws_caller_identity.current.account_id}:root"
        }
        Action   = "kms:*"
        Resource = "*"
      },
      {
        Sid    = "AllowAppRoleUsage"
        Effect = "Allow"
        Principal = {
          AWS = aws_iam_role.app.arn
        }
        Action = [
          "kms:Decrypt",
          "kms:DescribeKey",
          "kms:GenerateDataKey"
        ]
        Resource = "*"
      }
    ]
  })

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-main-kms"
  })
}

# Alias : nom humain pour la cle (plus simple a reperer dans la console)
resource "aws_kms_alias" "main" {
  name          = "alias/${local.name_prefix}-main"
  target_key_id = aws_kms_key.main.key_id
}

🟡 Dépendance circulaire apparente : aws_kms_key.main.policy référence aws_iam_role.app.arn, et les aws_iam_role_policy.app_kms référencent aws_kms_key.main.arn. Terraform résout l'ordre via le graphe — pas de problème tant qu'on ne met pas le rôle app dans le Principal avec kms:*.

Résultat attendu :

# + aws_kms_alias.main

Étape 4 : Security Groups (3) (30 min)

Objectif : créer les 3 SGs avec leurs règles en ressources séparées (syntaxe v5+ du provider AWS).

Contexte :

Depuis le provider AWS v5, les règles ingress/egress inline dans aws_security_group sont dépréciées au profit de ressources séparées :

  • aws_vpc_security_group_ingress_rule
  • aws_vpc_security_group_egress_rule

🔹 Pourquoi : les règles inline sont rafraîchies en bloc à chaque apply (risque de drift invisible). Les ressources séparées ont leur propre cycle de vie et un state ID dédié.

Règle clé : pour référencer un autre SG en source d'une règle, on utilise referenced_security_group_id (pas cidr_ipv4).

# sg.tf
#############################################
# SG ALB - accepte HTTPS et HTTP depuis Internet
#############################################
resource "aws_security_group" "alb" {
  name        = "${local.name_prefix}-alb"
  description = "SG pour l ALB public Nextcloud"
  vpc_id      = var.vpc_id

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-alb"
  })
}

# Ingress 443 depuis tout Internet (HTTPS)
resource "aws_vpc_security_group_ingress_rule" "alb_https" {
  security_group_id = aws_security_group.alb.id
  description       = "HTTPS from Internet"
  cidr_ipv4         = "0.0.0.0/0"
  from_port         = 443
  to_port           = 443
  ip_protocol       = "tcp"
}

# Ingress 80 depuis tout Internet (pour redirect 80 -> 443)
resource "aws_vpc_security_group_ingress_rule" "alb_http_redirect" {
  security_group_id = aws_security_group.alb.id
  description       = "HTTP from Internet (redirect to HTTPS)"
  cidr_ipv4         = "0.0.0.0/0"
  from_port         = 80
  to_port           = 80
  ip_protocol       = "tcp"
}

# Egress ALL (l ALB doit pouvoir atteindre les cibles)
resource "aws_vpc_security_group_egress_rule" "alb_all" {
  security_group_id = aws_security_group.alb.id
  description       = "Allow all egress"
  cidr_ipv4         = "0.0.0.0/0"
  ip_protocol       = "-1"
}

#############################################
# SG app - accepte uniquement depuis le SG ALB
#############################################
resource "aws_security_group" "app" {
  name        = "${local.name_prefix}-app"
  description = "SG pour les EC2 Nextcloud derriere l ALB"
  vpc_id      = var.vpc_id

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-app"
  })
}

# Ingress 80 depuis le SG ALB UNIQUEMENT (pas de CIDR)
resource "aws_vpc_security_group_ingress_rule" "app_http_from_alb" {
  security_group_id            = aws_security_group.app.id
  description                  = "HTTP from ALB only"
  referenced_security_group_id = aws_security_group.alb.id
  from_port                    = 80
  to_port                      = 80
  ip_protocol                  = "tcp"
}

# Egress ALL (dnf update, Docker pull, RDS, S3, Secrets Manager)
resource "aws_vpc_security_group_egress_rule" "app_all" {
  security_group_id = aws_security_group.app.id
  description       = "Allow all egress for updates Docker pull RDS S3"
  cidr_ipv4         = "0.0.0.0/0"
  ip_protocol       = "-1"
}

#############################################
# SG db - accepte uniquement PostgreSQL depuis SG app
#############################################
resource "aws_security_group" "db" {
  name        = "${local.name_prefix}-db"
  description = "SG pour RDS PostgreSQL Nextcloud"
  vpc_id      = var.vpc_id

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-db"
  })
}

# Ingress 5432 TCP depuis le SG app UNIQUEMENT
resource "aws_vpc_security_group_ingress_rule" "db_pg_from_app" {
  security_group_id            = aws_security_group.db.id
  description                  = "PostgreSQL from app only"
  referenced_security_group_id = aws_security_group.app.id
  from_port                    = 5432
  to_port                      = 5432

Étape 5 : Secrets Manager (2) + random_password (25 min)

Objectif : générer 2 mots de passe aléatoires et les stocker dans Secrets Manager (chiffrés par la KMS CMK).

Contexte :

  • RDS PostgreSQL n'accepte pas tous les caractères (/, @, ", espace). → special = false pour db_password.

  • Défaut AWS : 30 jours — le secret reste "soft deleted" pendant 30 jours et empêche de recréer un secret du même nom.
  • En dev : mettez 0 pour destroy immédiat. Sinon vous allez lutter à chaque terraform destroy suivi de terraform apply.

Troubleshooting spécifique sécurité

🟥 Erreur n°1 : KMS key brickée (key policy sans root)

Symptôme : après terraform apply, impossible d'utiliser la clé. Erreur AccessDeniedException: The ciphertext refers to a customer master key that does not exist...

Cause : vous avez écrit une key policy qui ne donne pas kms:* au root account.

Solution : toujours inclure le statement EnableRootAccountAccess avec Principal = { AWS = "arn:aws:iam::ACCOUNT_ID:root" } et Action = "kms:*". Si la clé est brickée, il n'y a aucun moyen de la récupérer : il faut attendre deletion_window_in_days et la supprimer. D'où l'importance des 10 jours en dev (pas 30).

# Verifier la key policy actuelle
aws kms get-key-policy --key-id alias/kolab-dev-main --policy-name default

🟠 Erreur n°2 : AccessDenied sur secretsmanager:GetSecretValue depuis l'EC2

Symptôme : dans /var/log/cloud-init-output.log sur l'EC2 :

An error occurred (AccessDeniedException) when calling the GetSecretValue operation:
User: arn:aws:sts::039497794217:assumed-role/kolab-dev-app/i-0abc... is not authorized to perform:
secretsmanager:GetSecretValue on resource: arn:aws:secretsmanager:eu-west-1:039497794217:secret:kolab-dev/db/password-XXXXX

Causes possibles :

  1. L'instance profile n'est pas attaché au launch template (Rôle 3 l'a oublié).
  2. La policy app_secrets manque un ARN (par exemple vous avez mis aws_secretsmanager_secret.db_password.arn mais pas aws_secretsmanager_secret.admin_password.arn).
  3. La policy KMS manque kms:Decrypt → GetSecretValue retourne bien le chiffré mais pas le clair.

Solution :

# 1. Verifier que le role est bien attache a l EC2
aws ec2 describe-instances --instance-ids i-0abc... \
  --query 'Reservations[0].Instances[0].IamInstanceProfile'

# 2. Verifier les policies attachees au role
aws iam list-role-policies --role-name kolab-dev-app
aws iam list-attached-role-policies --role-name kolab-dev-app

# 3. Simuler la policy
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::039497794217:role/kolab-dev-app \
  --action-names secretsmanager:GetSecretValue kms:Decrypt \
  --resource-arns <secret_arn>

🟠 Erreur n°3 : RDS ne démarre pas (kms_key_id mal référencé)

Symptôme : Rôle 4 lance l'apply et RDS reste en status failed avec InvalidParameterCombination: KMS key not found.

Cause : dans envs/dev/main.tf, le Rôle 4 a référencé module.security.kms_key_id au lieu de module.security.kms_key_arn. RDS veut un ARN complet.

Solution : vérifier dans envs/dev/main.tf que module.data reçoit bien :

module "data" {
  kms_key_arn = module.security.kms_key_arn   # <- ARN complet, pas .id
}

🟡 Erreur n°4 : tfsec AVD-AWS-xxxx "IAM policy allows wildcard action"

Symptôme : tfsec remonte :

Result #1 HIGH Resource 'aws_iam_role_policy.app_s3' defines a policy with wildcarded actions.
  Resource: "s3:*"
  Action should not be wildcarded.

Cause : vous avez écrit Action = "s3:*" ou Resource = "*" dans une policy inline.

Solution : scoper les actions et ressources. Remplacer :

# 🔴 Mauvais
actions   = ["s3:*"]
resources = ["*"]

# 🟢 Bon
actions = [
  "s3:GetObject",
  "s3:PutObject",
  "s3:DeleteObject",
  "s3:ListBucket",
]
resources = [
  var.s3_primary_bucket_arn,
  "${var.s3_primary_bucket_arn}/*",
]

🟥 Erreur n°5 : SG db ouvert à 0.0.0.0/0 sur 5432 (tfsec AVD-AWS-0107)

Symptôme :

CRITICAL Security group rule allows ingress from public internet on database port.

Cause : vous avez écrit cidr_ipv4 = "0.0.0.0/0" sur le SG db au lieu de referenced_security_group_id.

Solution :

# 🔴 Mauvais
resource "aws_vpc_security_group_ingress_rule" "db_pg" {
  security_group_id = aws_security_group.db.id
  cidr_ipv4         = "0.0.0.0/0"   # RDS exposee a Internet !
  from_port         = 5432
  to_port           = 5432
  ip_protocol       = "tcp"
}

# 🟢 Bon
resource "aws_vpc_security_group_ingress_rule" "db_pg_from_app" {
  security_group_id            = aws_security_group.db.id
  referenced_security_group_id = aws_security_group.app.id   # SG-to-SG
  from_port                    = 5432
  to_port                      = 5432
  ip_protocol                  = "tcp"
}

🟡 Erreur n°6 : ResourceExistsException au 2e apply après destroy

Symptôme : après un terraform destroy puis terraform apply rapide :

Error: creating Secrets Manager Secret: InvalidRequestException:
You can't create this secret because a secret with this name is already scheduled for deletion.

Cause : recovery_window_in_days = 30 par défaut → le secret reste en "pending deletion" 30 jours et bloque la création d'un nouveau secret du même nom.

Solution : mettre recovery_window_in_days = 0 en dev.

resource "aws_secretsmanager_secret" "db_password" {
  name                    = "${local.name_prefix}/db/password"
  recovery_window_in_days = var.environment == "dev" ? 0 : 30
}

Si vous êtes déjà coincé, forcer la suppression :

aws secretsmanager delete-secret \
  --secret-id kolab-dev/db/password \
  --force-delete-without-recovery \
  --region eu-west-1

🟠 Erreur n°7 : Nextcloud spinne à l'infini au boot (KMS policy incomplète)

Symptôme : l'EC2 démarre, le user_data s'exécute, mais Nextcloud ne répond pas. Dans /var/log/cloud-init-output.log :

An error occurred (AccessDeniedException) when calling the Decrypt operation:
The ciphertext refers to a customer master key that does not exist, does not exist in this region,
or you are not allowed to access.

Cause : la policy app_kms ne donne pas kms:Decrypt sur la CMK, ou la key policy KMS ne laisse pas le rôle app l'utiliser.

Solution : vérifier les 2 côtés de la relation :

  1. Côté IAM role (iam.tf) : policy app_kms avec actions = ["kms:Decrypt", "kms:DescribeKey"] + resources = [aws_kms_key.main.arn].
  2. Côté key policy (kms.tf) : statement AllowAppRoleUsage avec Principal = { AWS = aws_iam_role.app.arn } + Action = ["kms:Decrypt", "kms:DescribeKey", "kms:GenerateDataKey"].

🔷 Les deux policies doivent être alignées — KMS est un service "double-veto" : IAM + key policy, les deux doivent autoriser.

Conseils du formateur

🔹 Commencez par variables.tf et outputs.tf avant même de coder les ressources. Ça bloque le contrat — si vous changez les noms d'outputs en cours de route, vous cassez les 3 autres rôles.

🔹 Utilisez data "aws_iam_policy_document" plutôt que jsonencode ou HEREDOC. Plus lisible, valide la syntaxe IAM à la compilation, et génère le JSON optimisé.

🔹 Testez les policies en isolation avec aws iam simulate-principal-policy avant même que Nextcloud soit déployé. Vous pouvez déjà valider que le rôle a bien secretsmanager:GetSecretValue sur le bon ARN.

🔹 Schématisez vos flux réseau au brouillon avant de coder les SGs. 3 SGs = 6 règles — une erreur et c'est l'heure qui part en debug.

🔹 La KMS key policy est le chemin critique : si vous vous trompez, vous perdez 1h à attendre la deletion window. Copiez le pattern EnableRootAccountAccess depuis l'encadré "Rappel concept : KMS CMK + key policy" en haut de cette fiche — ne réinventez pas.

🔹 tfsec est votre ami, lancez-le après chaque fichier pendant le dev. Mieux vaut 6 passes de 10s que découvrir 15 erreurs à 16h30.

🔹 En production réelle, vous utiliseriez plutôt manage_master_user_password = true sur aws_db_instance (AWS gère rotation + stockage). Ici on fait la version manuelle pour comprendre le pattern — c'est pédagogique.

🔹 Les 2 policies managées AWS bonus (SSM + CloudWatch) sont hautement recommandées : SSM Session Manager vous sauvera la vie quand Nextcloud plantera et que vous n'aurez pas de SSH.

Récap

🟢 Ce que vous savez maintenant faire :

  • Créer une KMS CMK avec key policy correcte (root + app role)
  • Écrire 3 SGs avec les nouvelles ressources v5+ (ingress_rule / egress_rule)
  • Scoper des policies IAM aux ARN exacts (ni wildcard action, ni wildcard resource)
  • Générer et stocker des secrets via random_password + Secrets Manager
  • Lier rôle IAM → instance profile → EC2 sans access key

🟡 Pièges à retenir :

  • Key policy KMS sans root = clé brickée, attendez la deletion window
  • s3:* ou Resource = "*" = tfsec HIGH + perte de points
  • SG db avec cidr_ipv4 = "0.0.0.0/0" = faille critique
  • recovery_window_in_days = 30 en dev = bloqué au 2e apply
  • IAM seule n'est pas suffisante pour KMS — aligner key policy ET IAM policy

🔷 Points clés à retenir :

  • KMS : 1 CMK, rotation activée, alias explicite, key policy minimale mais complète
  • SG : référence SG-to-SG via referenced_security_group_id, jamais via CIDR
  • IAM : moindre privilège = ARN exacts, actions nommées
  • Secrets : Secrets Manager + KMS customer-managed, jamais en clair
