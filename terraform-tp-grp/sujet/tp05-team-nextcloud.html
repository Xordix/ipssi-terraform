TP05 — Projet final en équipe : Nextcloud sur AWS

Format : Projet d'équipe noté (4 à 5 participants) Prérequis : Jour 1 et Jour 2 validés — TP01 (bucket S3 sécurisé) et TP02 (VPC complet) terminés

Objectif pédagogique

Ce TP est la conclusion de la formation : vous allez déployer en équipe une application de production réelle sur AWS avec Terraform.

  1. Réutiliser les briques vues jusqu'ici (HCL, variables, VPC, S3, sécurité de base).
  2. Découvrir les concepts avancés nécessaires à un projet réel : modules organisés, backend S3, IAM instance profile, KMS, Secrets Manager, Load Balancer — tous expliqués dans les fiches rôles, pas besoin d'avoir fait J3/J4.
  3. Travailler en équipe sur un monorepo Terraform, avec des contrats d'interface figés entre modules.
  4. Déployer Nextcloud — le Dropbox/Drive open-source auto-hébergé, utilisé par des milliers d'organisations.
  5. Livrer un dossier de rendu noté (template + zip) évalué sur 20 points.

🔷 Ce TP est noté — voir le Template de rendu et le Barème en fin de document.

🔵 Un starter kit vous est fourni : corriges/jour5/tp05-starter/ contient l'arborescence complète, les interfaces de modules pré-écrites, et les providers déjà configurés. Vous n'avez qu'à remplir les corps des modules.

Cahier des charges fonctionnel

Contexte métier

"Kolab" est un cabinet d'avocats de 40 collaborateurs, spécialisé dans le droit des affaires. Jusqu'ici, les dossiers clients étaient partagés via une solution SaaS américaine, mais les contraintes RGPD + secret professionnel + souveraineté imposent de migrer vers une solution auto-hébergée dans l'UE.

La DSI a tranché : Nextcloud (auto-hébergé sur AWS Irlande) pour le partage de fichiers, chiffrement au repos et en transit, authentification interne.

Exigences fonctionnelles

# Exigence Priorité
F1 Interface web Nextcloud accessible en HTTPS 🔴 Bloquant
F2 Stockage des fichiers utilisateur chiffré et externalisé (pas sur l'EC2) 🔴 Bloquant
F3 Base de données PostgreSQL managée avec sauvegarde automatique quotidienne 🔴 Bloquant
F4 Administrateur initial pré-créé (compte admin, mot de passe généré aléatoirement) 🔴 Bloquant
F5 Tolérance à la panne du serveur applicatif (redémarrage auto) 🟠 Important
F6 Logs d'accès HTTP centralisés et conservés 30 jours 🟠 Important
F7 Infrastructure reproductible sur un 2e environnement (staging) 🟡 Bonus

Cahier des charges technique

Exigences techniques

# Exigence Priorité
T1 AWS région eu-west-1 (Irlande), conforme RGPD 🔴
T2 Terraform ≥ 1.10 (backend S3 natif avec use_lockfile) 🔴
T3 Code organisé en modules réutilisables + environnement envs/dev/ 🔴
T4 State S3 avec versioning + chiffrement KMS + locking natif 🔴
T5 Chiffrement KMS de toutes les ressources stateful (S3, RDS, Secrets) 🔴
T6 Pas de credentials AWS dans le code ou les EC2 — IAM Instance Profile uniquement 🔴
T7 Mots de passe (DB + admin Nextcloud) stockés dans AWS Secrets Manager, lus par l'EC2 au boot 🔴
T8 ALB public HTTPS (self-signed pour ce TP) → ASG d'EC2 privées 🔴
T9 RDS PostgreSQL Multi-AZ (db.t3.micro), subnet privé 🔴
T10 S3 primary storage Nextcloud + bucket de logs ALB séparés, chiffrés KMS 🔴
T11 Nextcloud dans un container Docker démarré par le user_data de l'EC2 🔴
T12 ASG min=1 max=2 desired=1 — auto-recovery simple (pas de scaling horizontal, limité à 1 instance pour éviter file locking sans Redis) 🟠
T13 VPC Endpoints S3 et Secrets Manager pour éviter le trafic via NAT 🟠
T14 tfsec : 0 erreur HIGH ou CRITICAL sur le code final 🟠
T15 Chaque module documenté avec terraform-docs 🟡 Bonus
T16 CI/CD GitHub Actions (plan on PR) 🟡 Bonus

Contraintes de budget

  • 🟡 Budget cible < 150 € pour la journée (NAT + ALB + RDS + EC2 t3.small).
  • 🟥 Destroy obligatoire en fin de journée — un environnement qui tourne la nuit coûte ~10 € par jour, soit ~300 €/mois.

Fichier docs/RENDU.md — template à remplir

Le template détaillé est dans tp05-team-nextcloud/template-rendu.md.

Les sections obligatoires sont :

  1. Identification équipe : membres + rôles attribués
  2. Résumé architecture : 5 lignes max + un schéma Mermaid
  3. Arbitrages techniques faits : 3 choix minimum avec justification (ex: pourquoi single EC2 au lieu de 2 ? pourquoi self-signed cert ?)
  4. Retour sur les interfaces : ce qui a fonctionné, ce qui a bloqué
  5. Tableau terraform plan : copie du résumé Plan: N to add, ...
  6. Preuve de succès : 5 captures d'écran obligatoires (cf. sous-section)
  7. Coût estimé via Infracost ou calcul manuel
  8. Rétrospective : 3 choses qui ont marché, 3 qui ont bloqué, 3 améliorations
  9. Per-rôle : chaque rôle ajoute un paragraphe "Ce que j'ai livré" + "Ce qui m'a surpris"

Nouveaux pour ce TP — tous expliqués dans les fiches rôles

1. Backend S3 pour le state (Rôle 1)

Ce que c'est : par défaut, Terraform stocke son terraform.tfstate localement. Pour travailler en équipe, on le met dans un bucket S3 avec un verrou pour éviter les applys simultanés.

À quoi ça sert : partage du state entre coéquipiers, historique des versions, chiffrement des secrets stockés dans le state.

Exemple minimal :

terraform {
  backend "s3" {
    bucket       = "tf-state-kolab-formation"
    key          = "envs/dev/terraform.tfstate"
    region       = "eu-west-1"
    encrypt      = true
    use_lockfile = true  # TF >= 1.10, plus besoin de DynamoDB
  }
}

🔷 Bootstrap "chicken-and-egg" : le bucket state est lui-même de l'infra AWS, mais on ne peut pas le créer via Terraform (où stockerait-on son state ?). On le crée une seule fois via un script bash AWS CLI, c'est le rôle du Rôle 1.

2. Modules avec interfaces figées (Rôles 1-5)

Ce que c'est : un module = un dossier .tf qui déclare variable (inputs) et output (sorties). L'appelant passe des valeurs et récupère des outputs.

Règles :

  • ⚫ Un module ne déclare JAMAIS de provider (c'est l'appelant qui le fait).
  • 🟢 Les variable sont son interface publique — on ne les modifie pas en cours de route.
  • 🟢 Les output sont ce qu'il expose à l'appelant.

Exemple d'appel :

module "networking" {
  source       = "../../modules/networking"
  project_name = "kolab"
  vpc_cidr     = "10.30.0.0/16"
}

module "compute" {
  source = "../../modules/compute"
  vpc_id = module.networking.vpc_id   # consomme l'output
}

3. IAM Instance Profile (Rôle 5 + Rôle 3)

Ce que c'est : un "bracelet de permissions" attaché à une instance EC2. Au lieu de stocker des clés AWS sur la machine (dangereux), on attache un rôle IAM, et le SDK AWS récupère automatiquement des credentials éphémères via l'IMDS (Instance Metadata Service).

À quoi ça sert : ici, les EC2 accèdent à S3 (stockage fichiers Nextcloud) et Secrets Manager (mots de passe) sans aucune clé en dur.

Exemple minimal :

resource "aws_iam_role" "app" {
  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = { Service = "ec2.amazonaws.com" }
      Action    = "sts:AssumeRole"
    }]
  })
}

resource "aws_iam_instance_profile" "app" {
  name = "nextcloud-app-profile"
  role = aws_iam_role.app.name
}

# Sur aws_instance :
#   iam_instance_profile = aws_iam_instance_profile.app.name

4. KMS Customer Managed Key (Rôle 5)

Ce que c'est : AWS KMS = service de gestion de clés de chiffrement. Une CMK (Customer Managed Key) = une clé dont vous gérez la policy. On l'utilise pour chiffrer les ressources stateful (S3, RDS, Secrets).

Règle critique — key policy : le compte root doit toujours avoir tous les droits sur la clé. Sinon la clé est "brickée" et il faut 7-30 jours pour la récupérer.

resource "aws_kms_key" "main" {
  description         = "CMK pour Nextcloud"
  enable_key_rotation = true
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Sid       = "EnableRootAccountAccess"
      Effect    = "Allow"
      Principal = { AWS = "arn:aws:iam::${data.aws_caller_identity.current.account_id}:root" }
      Action    = "kms:*"
      Resource  = "*"
    }]
  })
}

5. Secrets Manager (Rôle 5 + Rôle 3 + Rôle 4)

Ce que c'est : service AWS qui stocke des secrets (mots de passe, tokens) chiffrés avec KMS, avec rotation, versioning, et accès contrôlé par IAM.

À quoi ça sert : le mot de passe de la DB PostgreSQL + le mot de passe admin de Nextcloud sont générés aléatoirement, stockés ici, et lus par l'EC2 au boot via aws secretsmanager get-secret-value.

resource "random_password" "db" {
  length  = 24
  special = false  # RDS PG refuse certains caracteres
}

resource "aws_secretsmanager_secret" "db_password" {
  name       = "kolab-dev-db-password"
  kms_key_id = aws_kms_key.main.arn
}

resource "aws_secretsmanager_secret_version" "db_password" {
  secret_id     = aws_secretsmanager_secret.db_password.id
  secret_string = random_password.db.result
}

6. Application Load Balancer (Rôle 3)

Ce que c'est : un load balancer HTTP/HTTPS géré par AWS. Il écoute sur un port public (443), termine le TLS, et route les requêtes vers un target group (ici, l'EC2 Nextcloud).

Composants :

  • aws_lb : le load balancer lui-même
  • aws_lb_target_group : le groupe cible (une ou plusieurs instances)
  • aws_lb_target_group_attachment : relie une instance à un target group
  • aws_lb_listener : le listener sur un port (80 ou 443)
  • aws_acm_certificate : le certificat TLS (ici self-signed pour le TP)

7. user_data avec templatefile() (Rôle 3)

Ce que c'est : un script shell exécuté une seule fois au premier boot de l'EC2 (cloud-init). On interpole des variables Terraform dedans via templatefile().

Exemple :

user_data = templatefile("${path.module}/templates/nextcloud.sh.tftpl", {
  db_endpoint            = var.db_endpoint
  db_password_secret_arn = var.db_password_secret_arn
  s3_bucket              = var.s3_primary_bucket_name
})

Dans le template : ${db_endpoint} sera remplacé par la valeur. Le shell $VARIABLE doit être échappé en $$VARIABLE.

8. RDS PostgreSQL avec Multi-AZ (Rôle 4)

Ce que c'est : une base de données PostgreSQL managée par AWS. Multi-AZ = AWS maintient une réplique synchrone dans une autre AZ pour la haute disponibilité.

resource "aws_db_instance" "nextcloud" {
  engine             = "postgres"
  engine_version     = "16.4"
  instance_class     = "db.t3.micro"
  allocated_storage  = 20
  storage_encrypted  = true
  kms_key_id         = var.kms_key_arn
  multi_az           = true
  db_name            = "nextcloud"
  username           = "nextcloud"
  password           = data.aws_secretsmanager_secret_version.db_password.secret_string
  db_subnet_group_name   = aws_db_subnet_group.main.name
  vpc_security_group_ids = [var.db_security_group_id]
  skip_final_snapshot    = true  # dev uniquement
}

Pattern de user_data Nextcloud (extrait — à compléter par le Rôle 3)

#!/bin/bash
# user_data : recupere les secrets, lance Nextcloud en Docker.
set -euxo pipefail

# 1. Install Docker
dnf install -y docker
systemctl enable --now docker

# 2. Recuperer les secrets via IAM instance profile
DB_PASSWORD=$(aws secretsmanager get-secret-value --region eu-west-1 \
  --secret-id "${db_password_secret_arn}" --query SecretString --output text)

ADMIN_PASSWORD=$(aws secretsmanager get-secret-value --region eu-west-1 \
  --secret-id "${admin_password_secret_arn}" --query SecretString --output text)

# 3. Lancer Nextcloud en Docker
docker run -d --name nextcloud \
  --restart always \
  -p 80:80 \
  -e POSTGRES_HOST="${db_endpoint}" \
  -e POSTGRES_DB="${db_name}" \
  -e POSTGRES_USER="${db_username}" \
  -e POSTGRES_PASSWORD="$DB_PASSWORD" \
  -e NEXTCLOUD_ADMIN_USER=admin \
  -e NEXTCLOUD_ADMIN_PASSWORD="$ADMIN_PASSWORD" \
  -e OBJECTSTORE_S3_BUCKET="${s3_primary_bucket_name}" \
  -e OBJECTSTORE_S3_REGION=eu-west-1 \
  -e OBJECTSTORE_S3_USEPATH_STYLE=false \
  -e OBJECTSTORE_S3_SSL=true \
  -e OBJECTSTORE_S3_AUTOCREATE=false \
  -e OBJECTSTORE_S3_USE_EC2_IAM=true \
  -e NEXTCLOUD_TRUSTED_DOMAINS="${trusted_domain}" \
  nextcloud:30-apache
