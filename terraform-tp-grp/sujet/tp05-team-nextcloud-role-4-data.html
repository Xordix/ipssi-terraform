Rôle 4 — Data Engineer — TP05 Nextcloud

Durée : 6h (journée complète) Module livré : modules/data/ % de la note : 20 %

🔵 Starter kit : vous démarrez depuis corriges/jour5/tp05-starter/ qui contient déjà l'arborescence, les interfaces figées (variables.tf + outputs.tf) et les providers. Vous remplissez les fichiers .tf vides avec des TODO.

Votre mission

  • Construire l'étage données de Nextcloud : RDS PostgreSQL 16 Multi-AZ chiffré KMS + DB subnet group sur les 2 subnets privés DB.
  • Livrer 2 buckets S3 : le primary storage Nextcloud (où atterrissent les fichiers uploadés par les avocats) + le bucket d'access logs ALB (consommé par le Rôle 3).
  • Appliquer les bonnes pratiques TP01 sur chaque bucket : versioning (primary), chiffrement SSE-KMS, Block Public Access 4/4, bucket policy deny insecure transport, lifecycle sur les logs.
  • Consommer le mot de passe DB depuis Secrets Manager (créé par le Rôle 5) via data "aws_secretsmanager_secret_version".
  • Publier les outputs figés : db_endpoint, db_port, db_name, db_username, s3_primary_bucket_name/arn, s3_logs_bucket_name/arn.

🟤 Contexte Kolab : les fichiers du cabinet sont confidentiels (dossiers clients, pièces de procédure). Le stockage RDS + S3 doit être chiffré KMS avec la clé gérée par le Rôle 5. Le RDS est Multi-AZ (T9 du CdC) — si l'AZ-a tombe, bascule auto en AZ-b. Le bucket primary est versionné pour que la secrétaire du cabinet puisse restaurer un fichier écrasé par erreur via l'interface Nextcloud.

Concepts clés pour ce rôle

Déjà vus (J1 - J2) :

  • TP01 — Bucket S3 sécurisé (versioning, SSE, BPA, policy TLS) : vous réutilisez exactement ce pattern, étendu à SSE-KMS
  • Module 07 — Variables / outputs / data sources : 7 variables d'entrée, data "..." pour interroger AWS
  • Module 08 — VPC AWS : vous consommez private_db_subnet_ids (2 subnets sur 2 AZ)

Nouveaux pour ce rôle (ré-expliqués en contexte ci-dessous) :

  • RDS PostgreSQL Multi-AZ
  • DB subnet group
  • Bucket policies avancées (deny insecure transport, autoriser service AWS)
  • Secrets Manager — lecture via data source
  • Lifecycle rules S3 (transition Glacier, expiration)

🔵 Rappel concept : RDS Multi-AZ

RDS = service de base de données managée par AWS (PostgreSQL, MySQL, etc.). Multi-AZ = AWS maintient une réplique synchrone dans une 2e AZ. Si l'AZ primaire tombe, bascule automatique en 60-120 s sans action de votre part. Un aws_db_subnet_group liste les 2 subnets privés DB (un par AZ) — AWS choisit la primaire et la standby automatiquement.

resource "aws_db_instance" "nextcloud" {
  engine            = "postgres"
  engine_version    = "16.4"
  instance_class    = "db.t3.micro"
  storage_encrypted = true
  kms_key_id        = var.kms_key_arn
  multi_az          = true
  db_subnet_group_name   = aws_db_subnet_group.main.name
  vpc_security_group_ids = [var.db_security_group_id]
  skip_final_snapshot    = true  # dev uniquement
}

🔵 Rappel concept : lire un secret via data "aws_secretsmanager_secret_version"

Le Rôle 5 crée le secret db_password et vous passe son ARN en variable. Vous le lisez sans jamais l'écrire en clair dans le code, via une data source. Le mot de passe transite uniquement via le state (chiffré KMS par le backend S3).

data "aws_secretsmanager_secret_version" "db_password" {
  secret_id = var.db_password_secret_arn
}

# Puis, dans aws_db_instance :
password = data.aws_secretsmanager_secret_version.db_password.secret_string

🔵 Rappel concept : bucket policies avancées

En plus du Block Public Access du TP01, deux patterns de bucket policy production :

  • Deny insecure transport : refuse toute requête non-HTTPS via condition aws:SecureTransport = false
  • Allow service AWS : autorise un principal AWS (ex. aws_elb_service_account) à écrire les logs ALB

On utilise data "aws_iam_policy_document" pour écrire ces policies en HCL (plus lisible que le JSON brut).

🔵 Rappel concept : lifecycle rules S3

Une aws_s3_bucket_lifecycle_configuration définit des règles de transition/expiration automatiques des objets : par exemple "après 30 jours, basculer vers Glacier Instant Retrieval ; après 90 jours, supprimer". Idéal pour les logs ALB qui grossissent vite.

Étape 2 : versions.tf + locals (10 min)

Objectif : figer les versions et centraliser le préfixe de nommage.

Contexte : les noms de bucket S3 sont globalement uniques dans le monde entier. On suffixe avec l'account_id + random_pet pour éviter les collisions inter-équipes pendant le TP.

Exemple :

# versions.tf
terraform {
  required_version = ">= 1.10"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.70"
    }
    random = {
      source  = "hashicorp/random"
      version = "~> 3.6"
    }
  }
}

# main.tf (locals + data sources partages)

data "aws_caller_identity" "current" {}

resource "random_pet" "bucket_suffix" {
  length    = 2
  separator = "-"
}

locals {
  name_prefix = "${var.project_name}-${var.environment}"

  # Noms globalement uniques pour S3 (account_id + random_pet)
  primary_bucket_name = "${local.name_prefix}-primary-${data.aws_caller_identity.current.account_id}-${random_pet.bucket_suffix.id}"
  logs_bucket_name    = "${local.name_prefix}-alb-logs-${data.aws_caller_identity.current.account_id}-${random_pet.bucket_suffix.id}"

  common_tags = {
    Project     = var.project_name
    Environment = var.environment
    ManagedBy   = "terraform"
    Module      = "data"
  }
}

🔷 Point clé à vérifier : terraform init télécharge 2 providers (aws et random).

Étape 3 : Lecture du mot de passe depuis Secrets Manager (15 min)

Objectif : lire le mot de passe DB sans jamais l'écrire en clair dans le code.

Contexte : le Rôle 5 a créé le secret kolab-dev-db-password avec un password généré aléatoirement (random_password). Vous le lisez via data.aws_secretsmanager_secret_version — le mot de passe transite par le state Terraform (chiffré KMS côté backend), jamais par un fichier.

Exemple :

# rds.tf (partie 1 : lecture du secret)

# -----------------------------------------------------------------------------
# Lecture du mot de passe DB genere par le module security
# Le secret arn est passe en variable, Terraform resout le graphe de dependances.
# -----------------------------------------------------------------------------
data "aws_secretsmanager_secret_version" "db_password" {
  secret_id = var.db_password_secret_arn
}

🟡 Piège classique : si le Rôle 5 n'a pas encore pushé son code, terraform plan echoue avec ResourceNotFound. Normal — ce module dépend du Rôle 5. Dans envs/dev, Terraform résout le graphe et crée d'abord module.security puis module.data.

🔷 Point clé à vérifier : le mot de passe n'apparaît pas dans votre code — seulement l'ARN du secret.

Étape 4 : DB subnet group + RDS PostgreSQL Multi-AZ (40 min)

Objectif : déployer le RDS avec toutes les options de sécurité production.

Contexte : RDS Multi-AZ = AWS provisionne une réplique synchrone dans une 2e AZ. Si l'AZ primaire tombe, bascule automatique en 60-120s sans action de notre part. Le db_subnet_group liste les 2 subnets privés DB (un par AZ) — AWS choisit la primaire et la standby automatiquement.

Exemple :

# rds.tf (partie 2 : RDS)

# -----------------------------------------------------------------------------
# DB Subnet Group : RDS a besoin d au moins 2 subnets sur 2 AZ
# -----------------------------------------------------------------------------
resource "aws_db_subnet_group" "nextcloud" {
  name       = "${local.name_prefix}-db-subnets"
  subnet_ids = values(var.private_db_subnet_ids)

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-db-subnets"
  })
}

# -----------------------------------------------------------------------------
# Parameter Group : parametres PostgreSQL personnalises
# On force TLS cote serveur (rds.force_ssl = 1)
# -----------------------------------------------------------------------------
resource "aws_db_parameter_group" "nextcloud" {
  name   = "${local.name_prefix}-pg16"
  family = "postgres16"

  parameter {
    name  = "rds.force_ssl"
    value = "1"
  }

  parameter {
    name  = "log_statement"
    value = "ddl"  # logue CREATE/ALTER/DROP (audit)
  }

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-pg16-params"
  })
}

# -----------------------------------------------------------------------------
# Instance RDS PostgreSQL 16 Multi-AZ chiffree KMS
# -----------------------------------------------------------------------------
resource "aws_db_instance" "nextcloud" {
  identifier = "${local.name_prefix}-nextcloud"

  # Moteur
  engine         = "postgres"
  engine_version = var.db_engine_version

  # Taille compute + storage
  instance_class        = var.db_instance_class
  allocated_storage     = var.db_allocated_storage
  max_allocated_storage = var.db_max_allocated_storage
  storage_type          = "gp3"
  storage_encrypted     = true
  kms_key_id            = var.kms_key_arn

  # Base, identifiants
  db_name  = "nextcloud"
  username = "nextcloud"
  password = data.aws_secretsmanager_secret_version.db_password.secret_string

  # Reseau (prive uniquement, pas d acces public)
  db_subnet_group_name   = aws_db_subnet_group.nextcloud.name
  vpc_security_group_ids = [var.db_security_group_id]
  publicly_accessible    = false
  port                   = 5432

  # Haute dispo
  multi_az                = true
  backup_retention_period = 7
  backup_window           = "03:00-04:00"
  maintenance_window      = "sun:04:00-sun:05:00"

  # Upgrades
  auto_minor_version_upgrade = true
  apply_immediately          = false  # false en prod, ici peu importe

  # Logs vers CloudWatch
  enabled_cloudwatch_logs_exports = ["postgresql", "upgrade"]
  performance_insights_enabled    = true
  performance_insights_kms_key_id = var.kms_key_arn

  # Destruction (dev only)
  deletion_protection      = false  # 🟡 TRUE en prod obligatoire
  skip_final_snapshot      = true   # 🟡 FALSE en prod obligatoire
  delete_automated_backups = true

  # Parametres custom
  parameter_group_name = aws_db_parameter_group.nextcloud.name

  # IAM auth (bonus securite - ne remplace pas le mdp, s ajoute)
  iam_database_authentication_enabled = true

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-nextcloud-rds"
  })

  lifecycle {
    # Le password peut changer si le Role 5 regenere le secret.
    # Ignorer evite un recreate cyclique en cas de rotation.
    ignore_changes = [password]
  }
}

🟡 Pièges production :

  • deletion_protection = false + skip_final_snapshot = true = OK pour ce TP (destroy facile), JAMAIS en prod. En prod : true + true inversés, et snapshot obligatoire.
  • publicly_accessible = false : toujours. Même en dev, un RDS public est une faute grave (tfsec le remonte CRITICAL).
  • storage_encrypted = true + kms_key_id : obligatoire (T5 du CdC).

Étape 5 : Bucket S3 primary storage (30 min)

Objectif : bucket versionné chiffré KMS pour le stockage Nextcloud, avec BPA 4/4 et deny TLS.

Contexte : Nextcloud écrit chaque fichier uploadé ici via S3 API (option OBJECTSTORE_S3_* côté user_data). Les policies IAM qui permettent au rôle app d'écrire ici sont créées par le Rôle 5 — vous, vous sécurisez le bucket côté bucket policy (deny TLS false).

Exemple :

# s3.tf (partie 1 : primary)

# -----------------------------------------------------------------------------
# Bucket primary : stockage fichiers Nextcloud
# -----------------------------------------------------------------------------
resource "aws_s3_bucket" "primary" {
  bucket        = local.primary_bucket_name
  force_destroy = true  # 🟡 TP only - permet destroy avec objets presents

  tags = merge(local.common_tags, {
    Name = local.primary_bucket_name
    Role = "nextcloud-primary-storage"
  })
}

# Versioning : permet de restaurer un fichier ecrase
resource "aws_s3_bucket_versioning" "primary" {
  bucket = aws_s3_bucket.primary.id

  versioning_configuration {
    status = "Enabled"
  }
}

# Chiffrement SSE-KMS avec la CMK du module security
resource "aws_s3_bucket_server_side_encryption_configuration" "primary" {
  bucket = aws_s3_bucket.primary.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm     = "aws:kms"
      kms_master_key_id = var.kms_key_arn
    }
    # bucket_key reduit les couts KMS (mutualisation)
    bucket_key_enabled = true
  }
}

# Block Public Access 4/4 (jamais exposer les fichiers avocats !)
resource "aws_s3_bucket_public_access_block" "primary" {
  bucket = aws_s3_bucket.primary.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# -----------------------------------------------------------------------------
# Bucket policy : deny toute requete non HTTPS
# -----------------------------------------------------------------------------
data "aws_iam_policy_document" "primary_deny_insecure" {
  statement {
    sid    = "DenyInsecureTransport"
    effect = "Deny"

    principals {
      type        = "*"
      identifiers = ["*"]
    }

    actions = ["s3:*"]

    resources = [
      aws_s3_bucket.primary.arn,
      "${aws_s3_bucket.primary.arn}/*"
    ]

    condition {
      test     = "Bool"
      variable = "aws:SecureTransport"
      values   = ["false"]
    }
  }
}

resource "aws_s3_bucket_policy" "primary" {
  bucket = aws_s3_bucket.primary.id
  policy = data.aws_iam_policy_document.primary_deny_insecure.json

  # IMPORTANT : BPA doit etre pose AVANT la policy, sinon AWS refuse
  depends_on = [aws_s3_bucket_public_access_block.primary]
}

🟡 Piège force_destroy : en production jamais true. Ici, pour que terraform destroy supprime le bucket même avec des fichiers dedans en fin de TP, on met true — sinon vous perdez 30 min à vider le bucket à la main.

🔷 Point clé à vérifier : terraform plan affiche + aws_s3_bucket.primary, + aws_s3_bucket_versioning.primary, + aws_s3_bucket_server_side_encryption_configuration.primary, + aws_s3_bucket_public_access_block.primary, + aws_s3_bucket_policy.primary.

Étape 6 : Bucket S3 access logs ALB (30 min)

Objectif : bucket d'access logs ALB chiffré KMS, avec policy qui autorise ELB à écrire + lifecycle Glacier/expiration.

Contexte : l'ALB (créé par le Rôle 3) écrit ses logs bruts dans ce bucket. AWS utilise un compte canonique par région pour signer ces puts — on le récupère via data.aws_elb_service_account. Les logs ALB grossissent vite (1 Mo/jour par instance) : on les bascule en Glacier Instant Retrieval après 30 jours (moins cher) et on les supprime à 90 jours (cf. F6 du CdC = 30 jours conservation minimum, on en fait 90 avec transition).

Exemple :

# s3.tf (partie 2 : logs ALB)

# -----------------------------------------------------------------------------
# Data source : compte AWS canonique ELB pour la region courante
# -----------------------------------------------------------------------------
data "aws_elb_service_account" "main" {}

# -----------------------------------------------------------------------------
# Bucket logs ALB
# -----------------------------------------------------------------------------
resource "aws_s3_bucket" "logs" {
  bucket        = local.logs_bucket_name
  force_destroy = true  # 🟡 TP only

  tags = merge(local.common_tags, {
    Name = local.logs_bucket_name
    Role = "alb-access-logs"
  })
}

# Chiffrement SSE-KMS (meme CMK que le primary)
resource "aws_s3_bucket_server_side_encryption_configuration" "logs" {
  bucket = aws_s3_bucket.logs.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm     = "aws:kms"
      kms_master_key_id = var.kms_key_arn
    }
    bucket_key_enabled = true
  }
}

# Block Public Access 4/4
resource "aws_s3_bucket_public_access_block" "logs" {
  bucket = aws_s3_bucket.logs.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# -----------------------------------------------------------------------------
# Bucket policy : autorise le compte AWS ELB de la region a PutObject
# -----------------------------------------------------------------------------
data "aws_iam_policy_document" "logs" {
  # Statement 1 : autorise l ALB a ecrire ses logs
  statement {
    sid    = "AllowELBAccessLogs"
    effect = "Allow"

    principals {
      type        = "AWS"
      identifiers = [data.aws_elb_service_account.main.arn]
    }

    actions = ["s3:PutObject"]

    resources = [
      "${aws_s3_bucket.logs.arn}/alb/AWSLogs/${data.aws_caller_identity.current.account_id}/*"
    ]
  }

  # Statement 2 : deny tout le reste non HTTPS (defense en profondeur)
  statement {
    sid    = "DenyInsecureTransport"
    effect = "Deny"

    principals {
      type        = "*"
      identifiers = ["*"]
    }

    actions = ["s3:*"]

    resources = [
      aws_s3_bucket.logs.arn,
      "${aws_s3_bucket.logs.arn}/*"
    ]

    condition {
      test     = "Bool"
      variable = "aws:SecureTransport"
      values   = ["false"]
    }
  }
}

resource "aws_s3_bucket_policy" "logs" {
  bucket = aws_s3_bucket.logs.id
  policy = data.aws_iam_policy_document.logs.json

  depends_on = [aws_s3_bucket_public_access_block.logs]
}

# -----------------------------------------------------------------------------
# Lifecycle : logs -> Glacier IR a J+30, expires a J+90
# -----------------------------------------------------------------------------
resource "aws_s3_bucket_lifecycle_configuration" "logs" {
  bucket = aws_s3_bucket.logs.id

  rule {
    id     = "alb-logs-lifecycle"
    status = "Enabled"

    filter {
      prefix = "alb/"
    }

    # Transition vers Glacier Instant Retrieval apres 30 jours
    transition {
      days          = 30
      storage_class = "GLACIER_IR"
    }

    # Expiration apres 90 jours
    expiration {
      days = 90
    }

    # Les versions non-current expirent a 7 jours
    noncurrent_version_expiration {
      noncurrent_days = 7
    }

    # Nettoyage des multipart uploads incomplets (cout cache)
    abort_incomplete_multipart_upload {
      days_after_initiation = 7
    }
  }
}

🟤 Note sur le data source ELB service account : depuis 2022, AWS a introduit un service principal logdelivery.elasticloadbalancing.amazonaws.com dans certaines régions. En eu-west-1, le data source aws_elb_service_account marche toujours et c'est l'approche la plus lisible + compatible. Pour la prod en 2026+ sur nouvelles régions (ap-southeast-5, etc.), vérifier la doc AWS.

🟡 Piège ARN du statement : le prefix doit être exactement alb/AWSLogs/{account_id}/*. Si vous mettez juste ${aws_s3_bucket.logs.arn}/*, ça marche aussi, mais c'est moins précis. Le prefix alb/ doit matcher le prefix = "alb" côté ALB (étape 4 Rôle 3).

Conseils du formateur

Lancez le RDS en premier, les S3 ensuite. Le RDS prend 7-10 min à provisionner (Multi-AZ oblige). Pendant qu'il se crée, vous faites vos 2 buckets, la policy ELB, les lifecycle rules. Si vous faites l'inverse, vous attendez 10 min les bras croisés en fin de matinée avec un tfsec qui râle pour rien. Attaquez les deux en parallèle dans Terraform dès que possible — Terraform parallélise nativement jusqu'à 10 créations simultanées.

Le data source aws_secretsmanager_secret_version est le point de couplage critique avec le Rôle 5. Coordonnez-vous avec lui pendant le Kick-off : le nom du secret doit être convenu (convention : {project}-{env}-db-password), la policy KMS doit autoriser le service RDS, et le secret doit être créé AVANT votre apply. Le Platform Lead (Rôle 1) orchestre l'ordre des modules dans envs/dev/main.tf — en théorie, Terraform résout seul via les références de variables, mais si ça plante à plan, terraform plan -target=module.security puis -target=module.data débloque.

Ne trichez JAMAIS sur les paramètres de sécurité. Mettre publicly_accessible = true "pour tester depuis mon poste" est une faute. Si vous devez tester la connexion à RDS, vous le faites depuis une EC2 du Rôle 3 via SSM Session Manager + psql. Idem pour les buckets : si vous désactivez le Block Public Access "juste 5 min", vous avez 5 min pour qu'un scanner de rancon trouve votre bucket et vous fasse perdre vos 3 points de sécurité sur la note. Le formateur lance tfsec sur le dépôt à la remise — anticipez, pas une faille.
